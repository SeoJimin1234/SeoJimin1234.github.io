## 시스템 콜 (System Call)

### OS-001. 시스템 콜이 무엇인지 설명해 주세요.

시스템 콜(System Call)은 **사용자 프로그램(User Space)이 운영체제 커널(Kernel Space)의 기능을 사용하기 위해 요청하는 인터페이스**입니다.

사용자 프로그램은 보안상의 이유로 하드웨어 자원(디스크, 네트워크, 메모리 등)에 직접 접근할 수 없습니다. 따라서 파일 읽기/쓰기, 프로세스 생성, 네트워크 통신 등 커널 기능이 필요한 작업을 수행할 때 시스템 콜을 통해 커널에 요청합니다.

---

### OS-002. 우리가 사용하는 시스템 콜의 예시를 들어주세요.

| 분류 | 시스템 콜 예시 |
|------|----------------|
| 프로세스 제어 | `fork()`, `exec()`, `exit()`, `wait()` |
| 파일 관리 | `open()`, `close()`, `read()`, `write()` |
| 디바이스 관리 | `ioctl()`, `read()`, `write()` |
| 정보 유지 | `getpid()`, `time()`, `alarm()` |
| 통신 | `pipe()`, `socket()`, `send()`, `recv()` |

예를 들어 C언어에서 `printf()`를 호출하면 내부적으로 `write()` 시스템 콜이 호출됩니다.

---

### OS-003. 시스템 콜이 운영체제에서 어떤 과정으로 실행되는지 설명해 주세요.

1. 사용자 프로그램이 시스템 콜 함수를 호출합니다.
2. CPU는 **트랩(Trap) 명령**을 통해 유저 모드에서 커널 모드로 전환됩니다.
3. CPU는 **시스템 콜 번호**를 참조해 커널 내 **시스템 콜 테이블(Interrupt Descriptor Table)**에서 해당 핸들러를 찾습니다.
4. 커널이 요청된 서비스를 처리합니다.
5. 처리가 완료되면 결과값을 사용자 프로그램에 반환하고, 다시 **유저 모드**로 복귀합니다.

---

### OS-004. 시스템 콜의 유형에 대해 설명해 주세요.

시스템 콜은 기능에 따라 5가지 유형으로 분류됩니다.

- **프로세스 제어(Process Control)**: 프로세스 생성·종료·대기 (`fork`, `exit`, `wait`)
- **파일 조작(File Manipulation)**: 파일 열기·닫기·읽기·쓰기 (`open`, `close`, `read`, `write`)
- **장치 조작(Device Manipulation)**: 디바이스 요청 및 해제 (`ioctl`, `read`, `write`)
- **정보 유지(Information Maintenance)**: 시스템 정보 조회·설정 (`getpid`, `time`)
- **통신(Communication)**: 프로세스 간 통신 (`pipe`, `socket`, `mmap`)

---

### OS-005. 운영체제의 Dual Mode에 대해 설명해 주세요.

Dual Mode란 CPU가 동작하는 두 가지 모드입니다.

- **유저 모드(User Mode)**: 일반 사용자 프로그램이 실행되는 모드. 하드웨어 자원에 직접 접근 불가.
- **커널 모드(Kernel Mode)**: 운영체제가 실행되는 모드. 모든 하드웨어 자원에 직접 접근 가능.

CPU는 **모드 비트(Mode Bit)**로 현재 모드를 구분하며, 유저 모드에서는 특권 명령(Privileged Instruction)을 실행할 수 없습니다.

---

### OS-006. 왜 유저 모드와 커널 모드를 구분해야 하나요?

**보안과 안정성** 때문입니다. 만약 사용자 프로그램이 커널 모드로 직접 하드웨어에 접근할 수 있다면, 악의적인 프로그램이 운영체제나 다른 프로세스의 메모리를 무단으로 읽거나 수정할 수 있습니다. 또한 잘못된 접근으로 인해 시스템 전체가 다운될 수 있습니다.

Dual Mode를 통해 운영체제는 하드웨어 자원에 대한 **통제권**을 가지며, 사용자 프로그램이 시스템을 손상시키는 것을 방지합니다.

---

### OS-007. 서로 다른 시스템 콜을 어떻게 구분할 수 있을까요?

각 시스템 콜에는 고유한 **시스템 콜 번호(System Call Number)**가 부여되어 있습니다. 사용자 프로그램이 시스템 콜을 호출할 때 이 번호를 레지스터(보통 `eax` 또는 `rax`)에 저장하여 전달합니다.

커널은 이 번호를 **시스템 콜 테이블(System Call Table)**의 인덱스로 사용해 해당 핸들러 함수를 찾아 실행합니다. 예를 들어 리눅스에서 `read`는 번호 0, `write`는 번호 1, `open`은 번호 2입니다.

## 인터럽트 (Interrupt)

### OS-008. 인터럽트가 무엇인지 설명해 주세요.

인터럽트(Interrupt)는 **CPU가 현재 실행 중인 작업을 일시 중단하고 우선적으로 처리해야 할 이벤트를 처리하도록 하는 신호**입니다.

예를 들어 키보드 입력, 타이머 만료, I/O 완료 등이 발생하면 해당 장치가 CPU에 인터럽트 신호를 보냅니다. CPU는 현재 작업의 상태를 저장하고 인터럽트 처리기(ISR, Interrupt Service Routine)를 실행한 뒤, 기존 작업으로 복귀합니다.

---

### OS-009. 인터럽트는 어떻게 처리하나요?

1. 인터럽트 발생: 장치 또는 소프트웨어가 CPU에 인터럽트 신호를 전달합니다.
2. 현재 상태 저장: CPU는 현재 실행 중이던 프로세스의 레지스터 값과 PC(Program Counter)를 **스택**에 저장합니다.
3. ISR 실행: **인터럽트 벡터 테이블(Interrupt Vector Table)**에서 해당 인터럽트 번호에 대응하는 ISR 주소를 찾아 실행합니다.
4. 상태 복원: ISR 처리 완료 후, 저장해둔 상태를 복원하고 인터럽트가 발생하기 전 실행 중이던 위치로 돌아갑니다.

---

### OS-010. Polling 방식에 대해 설명해 주세요.

Polling은 **CPU가 주기적으로 장치의 상태를 직접 확인하는 방식**입니다. 인터럽트 방식과 달리 장치가 CPU에 신호를 보내지 않고, CPU가 능동적으로 장치 상태를 반복적으로 체크합니다.

- **장점**: 구현이 단순하며, 인터럽트 처리 오버헤드가 없어 응답이 예측 가능합니다.
- **단점**: 이벤트가 없을 때에도 CPU 자원을 낭비합니다(Busy Waiting).

실시간 시스템이나 이벤트가 매우 자주 발생하는 경우 Polling이 인터럽트보다 효율적일 수 있습니다.

---

### OS-011. HW / SW 인터럽트에 대해 설명해 주세요.

- **하드웨어 인터럽트(HW Interrupt)**: 외부 하드웨어 장치(키보드, 마우스, 타이머, 네트워크 카드 등)가 CPU에 보내는 신호입니다. 비동기적으로 발생합니다.
  - 예: 키보드 입력, I/O 완료, 타이머 인터럽트

- **소프트웨어 인터럽트(SW Interrupt, Trap)**: 실행 중인 프로그램이 의도적으로 발생시키는 인터럽트입니다. 시스템 콜 호출이 대표적이며, 동기적으로 발생합니다.
  - 예: 시스템 콜, 예외(Exception), 0으로 나누기 오류

---

### OS-012. 동시에 두 개 이상의 인터럽트가 발생하면 어떻게 처리해야 하나요?

**우선순위(Priority)** 기반으로 처리합니다.

- **인터럽트 우선순위**: 각 인터럽트에 우선순위가 부여되어 있으며, 높은 우선순위의 인터럽트가 먼저 처리됩니다.
- **마스킹(Masking)**: 현재 ISR 처리 중에 낮은 우선순위 인터럽트는 마스킹(차단)되고, 같거나 높은 우선순위의 인터럽트만 허용됩니다.
- **인터럽트 중첩**: 높은 우선순위의 인터럽트는 낮은 우선순위 ISR을 중단하고 먼저 처리될 수 있습니다(Nested Interrupt).

## 프로세스 (Process)

### OS-013. 프로세스가 무엇인가요?

프로세스(Process)는 **실행 중인 프로그램**입니다. 디스크에 저장된 프로그램(정적인 코드)이 메모리에 올라와 CPU에 의해 실행되는 동적인 상태를 의미합니다.

프로세스는 코드, 데이터, 스택, 힙 등의 메모리 공간과 PCB(Process Control Block)를 가지며, 운영체제가 관리하는 독립적인 실행 단위입니다.

---

### OS-014. 프로그램과 프로세스, 스레드의 차이에 대해 설명해 주세요.

| 구분 | 설명 |
|------|------|
| **프로그램** | 디스크에 저장된 정적인 실행 파일. 실행되기 전까지는 단순한 파일. |
| **프로세스** | 프로그램이 메모리에 올라와 실행되는 동적인 상태. 독립적인 메모리 공간(코드, 데이터, 스택, 힙)을 가짐. |
| **스레드** | 프로세스 내의 실행 단위. 같은 프로세스 내 스레드들은 코드, 데이터, 힙 영역을 공유하며, 각자 독립적인 스택과 레지스터를 가짐. |

프로세스는 서로 메모리를 공유하지 않아 IPC(Inter-Process Communication)가 필요한 반면, 같은 프로세스의 스레드들은 메모리를 공유해 통신이 간단하지만 동기화 문제에 주의해야 합니다.

---

### OS-015. PCB가 무엇인가요?

PCB(Process Control Block)는 **운영체제가 각 프로세스를 관리하기 위해 저장하는 자료구조**입니다. 프로세스가 생성될 때 만들어지고, 종료될 때 삭제됩니다.

PCB에 저장되는 정보:
- **프로세스 ID(PID)**: 프로세스 고유 식별자
- **프로세스 상태**: Ready, Running, Waiting 등
- **프로그램 카운터(PC)**: 다음에 실행할 명령어 주소
- **레지스터 값**: CPU 레지스터 상태
- **스케줄링 정보**: 우선순위, 스케줄링 큐 포인터
- **메모리 관리 정보**: 페이지 테이블, 세그먼트 테이블
- **I/O 상태 정보**: 열린 파일 목록 등

---

### OS-016. 스레드는 PCB를 갖고 있을까요?

스레드는 별도의 **TCB(Thread Control Block)**를 가집니다. TCB에는 스레드 ID, 스택 포인터, 프로그램 카운터, 레지스터 값 등이 저장됩니다.

다만 TCB는 PCB보다 가볍습니다. 스레드는 코드, 데이터, 힙 영역을 공유하므로 이를 따로 저장할 필요가 없기 때문입니다. 리눅스에서는 스레드도 내부적으로 프로세스처럼 `task_struct`로 관리하지만, 자원 공유 여부로 구분됩니다.

---

### OS-017. 리눅스에서 프로세스와 스레드는 각각 어떻게 생성될까요?

- **프로세스 생성**: `fork()` 시스템 콜을 사용합니다. 부모 프로세스를 **복제(Copy-on-Write)**하여 자식 프로세스를 만듭니다. 이후 `exec()`로 새 프로그램을 로드합니다.

- **스레드 생성**: `clone()` 시스템 콜을 사용합니다. `fork()`와 달리 메모리 공간, 파일 디스크립터, 신호 핸들러 등을 부모(프로세스)와 **공유**하도록 플래그를 설정합니다. 사용자 공간에서는 `pthread_create()` 함수로 추상화되어 사용됩니다.

즉 리눅스에서는 프로세스와 스레드 모두 `task_struct`로 관리되며, 자원 공유 여부가 차이입니다.

---

### OS-018. 자식 프로세스가 상태를 알리지 않고 죽거나, 부모 프로세스가 먼저 죽게 되면 어떻게 처리하나요?

- **좀비 프로세스(Zombie Process)**: 자식 프로세스가 종료되었지만 부모 프로세스가 `wait()`를 호출하지 않아 자식의 PCB가 제거되지 않은 상태입니다. 프로세스는 죽었지만 PCB가 남아 있어 PID를 점유합니다. 부모가 `wait()`를 호출하면 해소됩니다.

- **고아 프로세스(Orphan Process)**: 부모 프로세스가 먼저 종료된 경우입니다. 리눅스에서는 고아 프로세스를 **init 프로세스(PID 1)**가 입양하여 관리합니다. init은 주기적으로 `wait()`를 호출해 고아 프로세스를 정리합니다.

---

### OS-019. 리눅스에서 데몬 프로세스에 대해 설명해 주세요.

데몬(Daemon) 프로세스는 **백그라운드에서 지속적으로 실행되며 특정 서비스를 제공하는 프로세스**입니다. 사용자와 직접 상호작용하지 않으며, 터미널(제어 단말기)과 분리되어 있습니다.

- 예: `sshd`(SSH 서버), `httpd`(웹 서버), `cron`(작업 스케줄러)
- 특징:
  - 부모 프로세스로부터 분리(`fork()` 후 부모 종료)
  - 새로운 세션 생성(`setsid()`)
  - 제어 터미널 없음
  - 주로 PID 파일로 실행 여부 관리

---

### OS-020. 리눅스 프로세스 트리의 루트 노드 프로세스에 대해 설명해 주세요.

리눅스 부팅 시 커널이 가장 먼저 생성하는 프로세스는 **`init` 프로세스(PID 1)**입니다. 현대 리눅스(Ubuntu 15.04+, CentOS 7+)에서는 `systemd`가 init을 대체합니다.

- 모든 프로세스는 `init(systemd)`를 최상위 조상으로 하는 트리 구조를 형성합니다.
- 고아 프로세스를 입양하고 좀비 프로세스를 정리하는 역할을 합니다.
- 시스템 서비스 시작/종료, 런레벨(run level) 또는 타겟(target) 관리를 담당합니다.

---

## 프로세스 주소공간 (Process Address Space)

### OS-021. 프로세스 주소공간에 대해 설명해 주세요.

프로세스 주소공간은 **각 프로세스가 독립적으로 사용하는 가상 메모리 공간**으로, 일반적으로 아래 영역으로 구성됩니다.

| 영역 | 설명 |
|------|------|
| **코드(Text) 영역** | 실행할 프로그램의 기계어 코드. 읽기 전용. |
| **데이터(Data) 영역** | 초기화된 전역 변수, 정적 변수 저장. |
| **BSS 영역** | 초기화되지 않은 전역 변수, 정적 변수 저장. |
| **힙(Heap) 영역** | 동적으로 할당되는 메모리 (`malloc`, `new`). 낮은 주소에서 높은 주소 방향으로 증가. |
| **스택(Stack) 영역** | 함수 호출 시 생성되는 지역 변수, 매개변수, 반환 주소. 높은 주소에서 낮은 주소 방향으로 증가. |

---

### OS-022. 초기화하지 않은 변수들은 어디에 저장될까요?

초기화되지 않은 전역 변수와 정적 변수는 **BSS(Block Started by Symbol) 영역**에 저장됩니다.

BSS 영역은 프로그램 실행 전에 운영체제가 자동으로 0으로 초기화합니다. 실제 실행 파일에는 BSS 영역의 크기 정보만 저장되고 값은 포함되지 않으므로, 실행 파일 크기를 줄이는 효과가 있습니다.

---

### OS-023. Stack과 Heap의 크기는 매우 크다고 할 수 있을까요?

아닙니다. 주소공간 그림은 개념적인 표현이며, 실제 물리 메모리를 모두 사용하는 것은 아닙니다.

- **가상 메모리**: 프로세스는 가상 주소공간을 사용하므로 논리적으로는 큰 공간을 가진 것처럼 보이지만, 실제 물리 메모리는 필요한 만큼만 할당됩니다(Demand Paging).
- **스택 크기**: 일반적으로 시스템에 설정된 제한(리눅스 기본 8MB)이 있으며, 초과 시 **스택 오버플로우(Stack Overflow)**가 발생합니다.
- **힙 크기**: 실행 중 동적으로 결정되며, 가용 메모리가 허용하는 범위 내에서 확장 가능합니다.

---

### OS-024. Stack과 Heap 공간 중 접근 속도가 더 빠른 공간은 어디일까요?

**스택(Stack)**이 더 빠릅니다.

- **스택**: CPU가 스택 포인터(SP)를 통해 직접 관리하며, 메모리 할당/해제가 단순한 포인터 이동으로 이루어집니다. 또한 지역성이 높아 **캐시 히트율**이 높습니다.
- **힙**: `malloc`/`free` 등의 메모리 관리 함수가 내부적으로 복잡한 알고리즘(단편화 방지 등)을 수행하며, 접근 패턴이 불규칙해 캐시 효율이 낮습니다.

---

### OS-025. 다음과 같이 공간을 분할하는 이유가 있을까요?

공간 분할의 이유는 **접근 권한과 생명주기의 차이**를 관리하기 위함입니다.

- **코드 영역을 읽기 전용으로 분리**: 코드가 실수로 또는 악의적으로 수정되는 것을 방지합니다.
- **데이터/BSS를 분리**: 초기화 여부에 따라 실행 파일 크기 최적화와 초기화 방식을 구분합니다.
- **스택과 힙을 양끝에 배치**: 두 영역이 서로 반대 방향으로 성장하여 충돌 전까지 최대한 공간을 활용할 수 있습니다.
- **보안**: 각 영역에 적절한 실행/읽기/쓰기 권한을 부여해 버퍼 오버플로우 같은 공격을 완화합니다.

---

### OS-026. 스레드의 주소공간은 어떻게 구성되어 있을까요?

같은 프로세스 내의 스레드들은 **코드, 데이터, BSS, 힙 영역을 공유**합니다. 하지만 각 스레드는 **독립적인 스택**을 가집니다.

- **공유**: 코드(Text), 데이터(Data), BSS, 힙(Heap), 열린 파일 디스크립터, 신호 핸들러
- **독립**: 스택(Stack), 레지스터, 프로그램 카운터, 스레드 로컬 저장소(TLS)

스택을 독립적으로 가지는 이유는 각 스레드가 독립적인 함수 호출 흐름을 가져야 하기 때문입니다.

---

### OS-027. "스택" 영역과 "힙" 영역은 자료구조의 스택/힙과 연관이 있을까요?

- **스택(Stack) 영역**: 자료구조의 스택(LIFO)과 동작 방식이 동일합니다. 함수가 호출될 때 스택 프레임이 push되고, 함수가 반환될 때 pop됩니다. 이처럼 후입선출 방식으로 자동 관리됩니다.

- **힙(Heap) 영역**: 자료구조의 힙(최대/최소 힙 트리)과는 **직접적인 연관이 없습니다**. 메모리 힙은 동적 할당을 위한 자유 저장소(Free Store)로, "더미, 무더기"라는 의미에서 유래했습니다. 할당과 해제가 자유롭게 이루어지지만, 메모리 관리 알고리즘 내부에서 힙 자료구조를 활용할 수는 있습니다.

---

### OS-028. IPC의 Shared Memory 기법은 프로세스 주소공간의 어디에 들어가나요?

공유 메모리(Shared Memory)는 **힙(Heap)과 스택(Stack) 사이의 메모리 매핑 영역(Memory-Mapped Region)**에 위치합니다.

그 이유는 공유 메모리가 특정 프로세스의 코드/데이터가 아니라 **운영체제가 관리하는 특수한 영역**이기 때문입니다. `mmap()` 또는 `shmget()` 시스템 콜로 할당되며, 여러 프로세스의 주소공간에 동일한 물리 메모리 프레임을 매핑하는 방식으로 구현됩니다.

---

### OS-029. 스택과 힙 영역의 크기는 언제 결정되나요? 사용자가 수정할 수 있나요?

- **스택 크기**: 프로그램 실행 전 운영체제가 기본값으로 설정합니다(리눅스 기본 8MB). 컴파일 시 링커 옵션으로 변경하거나, 실행 중 `setrlimit()` 시스템 콜로 변경할 수 있습니다.
  - 사용자가 수정하는 방법: `ulimit -s [크기]` 명령어로 변경 가능합니다.

- **힙 크기**: 실행 중 `malloc`/`free`, `brk()`, `mmap()` 등을 통해 동적으로 결정됩니다. 시스템의 가용 메모리 범위 내에서 확장 가능하며, `ulimit -v`로 가상 메모리 제한을 설정할 수 있습니다.

## CPU 스케줄링 (CPU Scheduling)

### OS-030. 단기, 중기, 장기 스케줄러에 대해 설명해 주세요.

| 스케줄러 | 역할 | 주기 |
|---------|------|------|
| **장기(Long-term) 스케줄러** | 어떤 프로세스를 메모리에 올릴지 결정 (Job Scheduler). 디스크 → 메모리 | 드물게 실행 |
| **중기(Medium-term) 스케줄러** | 메모리가 부족할 때 프로세스를 디스크로 스왑 아웃하거나 다시 메모리로 올리는 결정 (Swapper) | 필요 시 실행 |
| **단기(Short-term) 스케줄러** | 준비 큐에서 어떤 프로세스를 CPU에 할당할지 결정 (CPU Scheduler) | 매우 자주 실행 (ms 단위) |

---

### OS-031. 현대 OS에는 단기, 중기, 장기 스케줄러를 모두 사용하고 있나요?

아닙니다. **현대 범용 OS(리눅스, Windows 등)는 장기 스케줄러를 거의 사용하지 않습니다.**

현대 OS에서는 사용자가 프로그램을 실행하면 바로 메모리에 올리는 방식을 사용합니다. 메모리가 부족한 경우 중기 스케줄러(스왑)를 통해 조절하며, CPU 할당은 단기 스케줄러가 전담합니다.

- **사용하는 것**: 단기 스케줄러 (필수), 중기 스케줄러 (스왑 필요 시)
- **거의 미사용**: 장기 스케줄러 (실시간 OS 등 일부 특수 시스템에서 사용)

---

### OS-032. 프로세스의 스케줄링 상태에 대해 설명해 주세요.

```
New → Ready → Running → Terminated
                ↕          ↓
             Waiting ←──────
             (Blocked)
                ↕
           Suspended
```

| 상태 | 설명 |
|------|------|
| **New** | 프로세스가 생성된 초기 상태 |
| **Ready** | CPU 할당을 기다리는 상태 (준비 큐에 존재) |
| **Running** | CPU를 할당받아 명령어를 실행 중인 상태 |
| **Waiting(Blocked)** | I/O 완료, 이벤트 등을 기다리는 상태 |
| **Terminated** | 실행이 완료된 상태 |
| **Suspended** | 중기 스케줄러에 의해 디스크로 스왑 아웃된 상태 |

---

### OS-033. preemptive/non-preemptive에서 존재할 수 없는 상태가 있을까요?

- **Non-preemptive(비선점)**: 프로세스가 자발적으로 CPU를 반납하거나 종료될 때만 스케줄링이 발생합니다. 따라서 타임 슬라이스 만료에 의한 **강제 선점이 없으므로**, Running → Ready 전환이 인터럽트에 의한 강제 선점으로는 발생하지 않습니다.

- **Preemptive(선점)**: 모든 상태 전환이 가능합니다.

엄밀히는 두 방식 모두 동일한 상태들이 존재하지만, Non-preemptive에서는 타이머 인터럽트에 의한 Running → Ready 전환이 일어나지 않는다는 점이 차이입니다.

---

### OS-034. Memory가 부족할 경우, Process는 어떠한 상태로 변화할까요?

메모리가 부족하면 중기 스케줄러가 개입하여 프로세스를 **Suspended(Swapped)** 상태로 전환합니다.

- **Suspended Ready**: Ready 상태였던 프로세스가 디스크로 스왑 아웃된 상태. 이벤트가 완료되었지만 메모리에 없음.
- **Suspended Waiting(Blocked)**: Waiting 상태였던 프로세스가 디스크로 스왑 아웃된 상태.

스왑 아웃된 프로세스의 메모리 내용은 스왑 영역(Swap Space, 디스크)에 저장되고, 메모리 여유가 생기면 다시 스왑 인(Swap In)되어 이전 상태로 복귀합니다.

---

## 컨텍스트 스위칭 (Context Switching)

### OS-035. 컨텍스트 스위칭 시에는 어떤 일들이 일어나나요?

컨텍스트 스위칭은 **현재 실행 중인 프로세스의 상태를 저장하고, 다음에 실행할 프로세스의 상태를 복원하는 과정**입니다.

1. **현재 프로세스 상태 저장**: 레지스터 값(PC, SP 등)을 현재 프로세스의 PCB에 저장합니다.
2. **현재 프로세스 상태 변경**: Running → Ready 또는 Waiting으로 전환합니다.
3. **다음 프로세스 선택**: 스케줄러가 실행할 프로세스를 선택합니다.
4. **다음 프로세스 상태 복원**: 선택된 프로세스의 PCB에서 레지스터 값을 복원합니다.
5. **실행 재개**: CPU가 복원된 PC부터 명령어를 실행합니다.

컨텍스트 스위칭 자체는 **오버헤드**로, 이 시간 동안 CPU는 유용한 작업을 수행하지 않습니다.

---

### OS-036. 프로세스와 스레드는 컨텍스트 스위칭이 발생했을 때 어떤 차이가 있을까요?

- **프로세스 간 컨텍스트 스위칭**: 메모리 주소공간이 다르므로 **페이지 테이블, TLB 플러시** 등 메모리 컨텍스트도 교체해야 합니다. 오버헤드가 큽니다.

- **스레드 간 컨텍스트 스위칭**: 같은 프로세스 내 스레드는 메모리 주소공간을 공유하므로 페이지 테이블, TLB를 교체할 필요가 없습니다. **스택과 레지스터만 교체**하면 되므로 프로세스 스위칭보다 훨씬 빠릅니다.

---

### OS-037. 컨텍스트 스위칭이 발생할 때, 기존 프로세스 정보는 커널 스택에 어떠한 형식으로 저장되나요?

커널 스택에는 **스택 프레임(Stack Frame)** 형태로 저장됩니다.

구체적으로는 하드웨어가 자동으로 저장하는 부분과 OS가 저장하는 부분으로 나뉩니다:

1. **하드웨어(CPU)가 자동 저장**: 인터럽트/예외 발생 시 SS, RSP, RFLAGS, CS, RIP(PC)를 커널 스택에 자동으로 push합니다.
2. **OS(ISR)가 저장**: 나머지 범용 레지스터(RAX, RBX, RCX 등)를 커널 스택에 push합니다.

이 모든 값이 PCB 또는 커널 스택에 저장된 뒤, `task_struct` 내부의 `thread` 구조체에 스택 포인터가 기록됩니다.

---

### OS-038. 컨텍스트 스위칭은 언제 일어날까요?

- **타이머 인터럽트**: Time Slice(퀀텀)가 만료되었을 때 (선점 스케줄링)
- **시스템 콜**: I/O 요청 등으로 프로세스가 Waiting 상태로 전환될 때
- **인터럽트**: 하드웨어 인터럽트 처리 후 다른 프로세스로 전환될 때
- **프로세스 종료**: 현재 프로세스가 `exit()`을 호출했을 때
- **우선순위 변경**: 더 높은 우선순위의 프로세스가 Ready 상태가 되었을 때 (선점 스케줄링)

---

## 스케줄링 알고리즘 (Scheduling Algorithms)

### OS-039. 프로세스 스케줄링 알고리즘에는 어떤 것들이 있나요?

| 알고리즘 | 설명 | 선점 여부 |
|---------|------|----------|
| **FCFS(FIFO)** | 먼저 도착한 순서대로 처리 | 비선점 |
| **SJF** | 가장 짧은 CPU 버스트 먼저 처리 | 비선점 |
| **SRTF** | SJF의 선점 버전, 남은 시간이 짧은 것 우선 | 선점 |
| **Priority** | 우선순위가 높은 프로세스 먼저 처리 | 선점/비선점 |
| **RR(Round Robin)** | 일정 Time Slice씩 돌아가며 처리 | 선점 |
| **MLFQ** | 여러 레벨의 큐를 사용, 동적으로 우선순위 조정 | 선점 |
| **Lottery** | 티켓을 무작위로 추첨하여 CPU 할당 | 선점 |

---

### OS-040. RR을 사용할 때, Time Slice에 따른 trade-off를 설명해 주세요.

- **Time Slice가 너무 짧은 경우**: 컨텍스트 스위칭이 매우 자주 발생하여 **오버헤드가 증가**합니다. 실제 작업보다 스위칭에 더 많은 시간이 소요될 수 있습니다.

- **Time Slice가 너무 긴 경우**: 사실상 FCFS와 비슷해져 **응답 시간이 길어집니다**. 짧은 작업도 긴 Time Slice가 끝날 때까지 기다려야 합니다.

- **적절한 Time Slice**: 일반적으로 컨텍스트 스위칭 시간의 10~100배 수준(보통 10~100ms)으로 설정합니다. 응답성과 처리량의 균형을 맞추는 것이 핵심입니다.

---

### OS-041. 싱글 스레드 CPU에서 상시로 돌아가야 하는 프로세스가 있다면, 어떤 스케줄링 알고리즘을 사용하는 것이 좋을까요?

**우선순위 기반 스케줄링(Priority Scheduling)**에서 해당 프로세스에 최고 우선순위를 부여하는 것이 좋습니다.

또는 **실시간 스케줄링(Real-time Scheduling)**인 **EDF(Earliest Deadline First)**나 **RM(Rate Monotonic)**을 사용할 수 있습니다.

이유: RR이나 MLFQ는 모든 프로세스에게 CPU 시간을 공평하게 나누기 때문에, 상시 실행이 필요한 프로세스가 CPU를 빼앗길 수 있습니다. 우선순위 기반으로 가장 높은 우선순위를 부여하면 항상 CPU를 선점할 수 있습니다.

---

### OS-042. 동시성과 병렬성의 차이에 대해 설명해 주세요.

- **동시성(Concurrency)**: **하나의 CPU**가 여러 작업을 **빠르게 번갈아 가며** 처리하여 마치 동시에 실행되는 것처럼 보이는 것입니다. 실제로는 한 번에 하나의 작업만 실행됩니다.
  - 예: 싱글 코어 CPU에서의 멀티태스킹

- **병렬성(Parallelism)**: **여러 CPU(코어)**가 **실제로 동시에** 여러 작업을 처리하는 것입니다.
  - 예: 멀티코어 CPU에서 각 코어가 다른 스레드를 실행

동시성은 "논리적 동시 실행"이고, 병렬성은 "물리적 동시 실행"입니다.

---

### OS-043. Multi-level Feedback Queue는 어떤 문제점들을 해결하나요?

MLFQ(Multi-Level Feedback Queue)는 다음 문제들을 해결합니다.

- **SJF/SRTF의 문제**: CPU 버스트 시간을 미리 알 수 없습니다. MLFQ는 실행 이력을 보고 동적으로 우선순위를 조정하여 짧은 작업을 우선 처리합니다.
- **Priority 스케줄링의 기아(Starvation)**: 낮은 우선순위 프로세스가 계속 기다리는 문제를 **Aging(시간이 지날수록 우선순위 상향)**으로 해결합니다.
- **RR의 비효율**: CPU 집약적(CPU-bound) 작업과 I/O 집약적(I/O-bound) 작업을 구분하지 못합니다. MLFQ는 CPU를 오래 사용하면 낮은 우선순위 큐로 이동시켜 I/O-bound 작업이 빠르게 처리되도록 합니다.

---

### OS-044. FIFO 스케줄러는 어떤 시나리오에 사용하면 좋을까요?

FIFO(FCFS)는 다음 상황에 유용합니다.

- **배치 처리 시스템(Batch System)**: 응답 시간보다 처리량이 중요하고, 모든 작업의 우선순위가 동일한 경우
- **모든 프로세스의 실행 시간이 비슷한 경우**: Convoy Effect(짧은 작업이 긴 작업 뒤에 밀리는 현상)가 발생하지 않아 효율적입니다.
- **구현 단순성이 중요한 경우**: FIFO는 구현이 매우 간단합니다.
- **임베디드/실시간 시스템**: 단일 작업만 실행하거나 우선순위가 명확히 정해진 경우

---

### OS-045. 스레드는 다른 방식으로 스케줄링을 하나요?

커널 스레드는 **프로세스와 동일한 방식으로 스케줄링**됩니다. 리눅스 등 현대 OS에서는 스케줄링의 기본 단위가 프로세스가 아닌 **태스크(task, 스레드)**입니다.

즉, 스케줄러는 프로세스 단위가 아니라 **스레드(태스크) 단위**로 CPU를 할당합니다. 따라서 같은 프로세스의 스레드들도 서로 독립적으로 스케줄링됩니다.

---

### OS-046. 유저 스레드와 커널 스레드의 스케줄링 알고리즘은 똑같을까요?

다릅니다.

- **커널 스레드**: OS 스케줄러가 직접 관리합니다. OS의 스케줄링 알고리즘(CFS, Priority 등)이 적용됩니다.

- **유저 스레드**: 사용자 공간의 스레드 라이브러리(Green Thread, Goroutine 등)가 관리합니다. OS는 유저 스레드의 존재를 모르며, 라이브러리가 커널 스레드(또는 프로세스) 위에서 자체적인 스케줄링 알고리즘을 구현합니다.

유저 스레드는 컨텍스트 스위칭 비용이 낮지만, I/O 블로킹 시 해당 커널 스레드 전체가 블록되는 문제가 있습니다.

---

## 프로세스 동기화 (Process Synchronization)

### OS-047. 뮤텍스와 세마포어의 차이점은 무엇인가요?

| 구분 | 뮤텍스(Mutex) | 세마포어(Semaphore) |
|------|--------------|-------------------|
| **값** | 0 또는 1 (잠금/잠금 해제) | 0 이상의 정수 |
| **소유권** | Lock을 획득한 스레드만 해제 가능 | 어떤 스레드든 signal 가능 |
| **용도** | 상호 배제(Mutual Exclusion) | 자원 접근 수 제한, 동기화 신호 |
| **예시** | 화장실 열쇠 1개 | 주차장 남은 자리 수 |

---

### OS-048. 이진 세마포어와 뮤텍스의 차이에 대해 설명해 주세요.

이진 세마포어(Binary Semaphore)는 값이 0 또는 1인 세마포어로, 겉보기엔 뮤텍스와 유사합니다. 그러나 핵심 차이가 있습니다.

- **소유권(Ownership)**: 뮤텍스는 Lock을 획득한 스레드만 해제할 수 있습니다. 이진 세마포어는 어떤 스레드든 signal(V 연산)을 호출해 해제할 수 있습니다.
- **용도**: 뮤텍스는 상호 배제에, 이진 세마포어는 동기화 신호 전달(예: 생산자-소비자)에 더 적합합니다.
- **재귀 잠금**: 일부 뮤텍스 구현은 같은 스레드가 재귀적으로 Lock을 획득할 수 있습니다(Reentrant Mutex).

---

### OS-049. Spin Lock의 장단점과 개선 방법은 무엇인가요?

**Spin Lock**은 Lock을 획득할 때까지 CPU를 점유하며 반복적으로 Lock 상태를 확인(Busy Waiting)하는 방식입니다.

- **장점**:
  - Lock 대기 시간이 매우 짧을 경우 컨텍스트 스위칭 없이 빠르게 Lock 획득 가능
  - 멀티코어 환경에서 효율적 (Lock 보유 스레드가 다른 코어에서 실행 중)
  
- **단점**:
  - CPU를 낭비합니다(Busy Waiting). 단일 코어에서는 Lock을 보유한 스레드가 실행될 수 없어 교착 상태가 발생할 수 있음
  - Lock 대기 시간이 길면 매우 비효율적

- **해결 방법**: 대기 시간이 길 것으로 예상될 때는 **슬리핑 뮤텍스(Sleeping Mutex)**를 사용합니다. 일정 횟수 spin 후에도 Lock을 얻지 못하면 스레드를 Waiting 상태로 전환하는 **Adaptive Mutex**도 있습니다.

---

### OS-050. 뮤텍스/세마포어의 커널 관리 방식 장단점과 개선 방법은?

뮤텍스와 세마포어는 커널이 관리하므로 Lock/Unlock 시마다 **시스템 콜**이 발생합니다.

- **장점**: 커널이 관리하므로 안전하고 정확한 동기화가 보장됩니다. Lock을 기다리는 스레드를 Waiting 상태로 전환해 CPU를 낭비하지 않습니다.

- **단점**: 시스템 콜 오버헤드(유저 모드 ↔ 커널 모드 전환)가 발생합니다. 경쟁이 없을 때도 시스템 콜 비용이 들어 성능 저하가 발생합니다.

- **해결 방법**: **Futex(Fast Userspace Mutex)**. 경쟁이 없는 경우(Lock이 비어 있는 경우)에는 **유저 공간에서 원자적 연산**으로만 처리하고, 경쟁이 발생하는 경우에만 커널을 호출합니다. 리눅스의 `pthreads` 뮤텍스가 이 방식을 사용합니다.

---

### OS-051. Deadlock에 대해 설명해 주세요.

데드락(Deadlock, 교착 상태)은 **두 개 이상의 프로세스가 서로 상대방이 가진 자원을 기다리며 무한히 대기하는 상태**입니다.

예시: 프로세스 A가 자원 1을 가지고 자원 2를 기다리고, 프로세스 B가 자원 2를 가지고 자원 1을 기다리면 둘 다 영원히 진행되지 못합니다.

---

### OS-052. Deadlock이 발생하기 위한 4가지 조건에 대해 설명해 주세요.

데드락은 다음 4가지 조건이 **동시에** 만족될 때 발생합니다.

1. **상호 배제(Mutual Exclusion)**: 자원은 한 번에 하나의 프로세스만 사용할 수 있습니다.
2. **점유와 대기(Hold and Wait)**: 자원을 보유한 채로 다른 자원을 기다립니다.
3. **비선점(No Preemption)**: 프로세스가 보유한 자원을 강제로 빼앗을 수 없습니다.
4. **순환 대기(Circular Wait)**: 프로세스들이 순환 형태로 서로의 자원을 기다립니다.

---

### OS-053. 3가지만 충족하면 왜 Deadlock이 발생하지 않을까요?

4가지 조건은 데드락의 **필요 조건**입니다. 4가지 중 하나라도 만족하지 않으면 데드락은 발생할 수 없습니다.

예를 들어:
- **순환 대기**가 없으면: 자원 그래프에 사이클이 없으므로 대기가 무한히 이어지지 않습니다.
- **비선점**이 없으면: 자원을 강제로 빼앗을 수 있으므로 교착 상태가 해소됩니다.
- **점유와 대기**가 없으면: 필요한 자원을 모두 한 번에 요청하거나, 자원이 없으면 보유한 것도 반납하므로 순환 대기가 형성되지 않습니다.

---

### OS-054. Deadlock을 어떤 방식으로 예방할 수 있을까요?

4가지 조건 중 하나를 제거하는 방식입니다.

| 조건 제거 | 방법 |
|----------|------|
| **상호 배제 제거** | 자원을 공유 가능하게 만들기 (읽기 전용 파일 등). 모든 자원에 적용하기 어려움 |
| **점유와 대기 제거** | 필요한 자원을 모두 한 번에 요청하거나, 자원 요청 시 보유 자원을 모두 반납 |
| **비선점 제거** | 자원을 강제로 빼앗을 수 있게 허용 (CPU, 메모리는 가능, 프린터 등은 어려움) |
| **순환 대기 제거** | 자원에 번호를 매기고 오름차순으로만 요청 (Resource Ordering) |

또는 **회피(Avoidance)**: 은행가 알고리즘(Banker's Algorithm)으로 안전 상태를 유지합니다.
또는 **탐지 및 회복(Detection & Recovery)**: 주기적으로 데드락을 탐지하고, 프로세스를 종료하거나 자원을 선점하여 복구합니다.

---

### OS-055. 왜 현대 OS는 Deadlock을 처리하지 않을까요?

현대 범용 OS(리눅스, Windows 등)는 데드락을 **무시(Ignore)**하는 정책인 **타조 알고리즘(Ostrich Algorithm)**을 주로 사용합니다. 이유는 다음과 같습니다.

- **발생 빈도가 낮습니다**: 일반적인 사용 환경에서 데드락은 드물게 발생합니다.
- **처리 비용이 큽니다**: 데드락 예방/회피는 자원 활용률을 낮추거나 복잡한 알고리즘을 요구합니다.
- **사용자 책임**: 응용 프로그램 개발자가 데드락이 발생하지 않도록 설계하는 것이 더 현실적입니다.
- **대안 존재**: 데드락이 발생하면 사용자가 프로세스를 강제 종료(kill)하거나 재시작하는 방식으로 해결합니다.

---

### OS-056. Wait Free와 Lock Free를 비교해 주세요.

둘 다 **락을 사용하지 않는(Lock-Free) 동시성 기법**이지만 보장 수준이 다릅니다.

| 구분 | Lock Free | Wait Free |
|------|-----------|-----------|
| **정의** | 시스템 전체적으로 최소 하나의 스레드가 진전(progress)을 이룸을 보장 | 모든 스레드가 유한한 시간 내에 반드시 완료됨을 보장 |
| **기아(Starvation)** | 특정 스레드가 영원히 진전을 못 할 수 있음 | 기아 없음 (모든 스레드 보장) |
| **구현 난이도** | 상대적으로 낮음 | 매우 높음 |
| **성능** | 일반적으로 우수 | 구현 복잡도로 인해 오버헤드 가능 |

CAS(Compare-And-Swap) 같은 원자적 연산을 사용합니다. Wait Free는 Lock Free보다 강한 조건입니다.

## 컴파일 (Compilation)

### OS-057. 프로그램이 컴파일되어 실행되는 과정을 간략하게 설명해 주세요.

```
소스 코드(.c)
    ↓ 전처리(Preprocessing)
전처리된 코드(.i) — #include, #define 처리
    ↓ 컴파일(Compilation)
어셈블리 코드(.s)
    ↓ 어셈블(Assembly)
오브젝트 파일(.o) — 기계어 코드, 심볼 테이블
    ↓ 링킹(Linking)
실행 파일(executable) — 라이브러리 연결
    ↓ 로딩(Loading)
메모리 적재 — 프로세스 실행
```

---

### OS-058. 링커와 로더의 차이에 대해 설명해 주세요.

- **링커(Linker)**: 여러 **오브젝트 파일(.o)**과 **라이브러리**를 하나의 실행 파일로 합치는 역할을 합니다. 심볼 참조(함수 호출, 변수 참조)를 실제 주소로 연결합니다.
  - **정적 링킹(Static Linking)**: 라이브러리 코드를 실행 파일에 포함
  - **동적 링킹(Dynamic Linking)**: 실행 시 `.so`(리눅스) 또는 `.dll`(Windows) 파일을 참조

- **로더(Loader)**: 실행 파일을 **메모리에 적재**하고 실행을 시작하는 역할을 합니다.
  - 코드, 데이터를 메모리의 적절한 위치에 복사
  - 동적 링킹의 경우 실행 시 라이브러리를 메모리에 매핑
  - PC(Program Counter)를 프로그램 시작 주소로 설정

---

### OS-059. 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.

| 구분 | 컴파일 언어 | 인터프리터 언어 |
|------|------------|----------------|
| **실행 방식** | 소스 코드를 실행 전 기계어로 완전 변환 | 소스 코드를 한 줄씩 읽으며 즉시 실행 |
| **실행 속도** | 빠름 (이미 기계어) | 느림 (실행 시 변환 오버헤드) |
| **오류 발견** | 컴파일 시 발견 | 실행 시 발견 |
| **이식성** | 플랫폼별로 재컴파일 필요 | 인터프리터만 있으면 어디서든 실행 가능 |
| **예시** | C, C++, Go, Rust | Python, Ruby, JavaScript(전통적) |

### OS-060. JIT에 대해 설명해 주세요.

JIT(Just-In-Time) 컴파일은 **인터프리터 방식의 단점을 보완하기 위해, 실행 중에 자주 실행되는 코드(Hot Spot)를 즉석에서 기계어로 컴파일**하는 기술입니다.

동작 방식:
1. 처음에는 바이트코드를 인터프리터로 실행합니다.
2. 자주 실행되는 코드를 감지(Profiling)합니다.
3. 해당 코드를 기계어로 컴파일하여 캐시에 저장합니다.
4. 이후 동일 코드는 기계어로 직접 실행합니다.

Java의 HotSpot JVM, JavaScript V8 엔진, Python의 PyPy 등에서 사용됩니다.

### OS-061. C, Java, Python의 컴파일 및 실행 과정 차이를 설명해 주세요.

**C 언어**:
```
소스 코드(.c) → [전처리 → 컴파일 → 어셈블 → 링킹] → 기계어 실행 파일 → OS에서 직접 실행
```

**Java**:
```
소스 코드(.java) → [javac 컴파일] → 바이트코드(.class) → JVM(인터프리터 + JIT) → 실행
```
플랫폼 독립적인 바이트코드를 JVM이 실행합니다. "Write Once, Run Anywhere"

**Python**:
```
소스 코드(.py) → [CPython 컴파일] → 바이트코드(.pyc) → Python 인터프리터 → 실행
```
Java와 유사하게 바이트코드를 생성하지만, JIT 없이 기본적으로 인터프리터 방식으로 실행됩니다(CPython 기준).

---

### OS-062. CPython, Jython, PyPy의 차이를 설명해 주세요.

| 구현체 | 언어 | 실행 방식 | 특징 |
|--------|------|----------|------|
| **CPython** | C | 인터프리터(바이트코드) | Python 표준 구현체. GIL 존재. |
| **Jython** | Java | JVM 위에서 실행 | Java 클래스와 상호 운용 가능. Java 라이브러리 사용 가능. |
| **PyPy** | Python(RPython) | JIT 컴파일 | CPython보다 평균 4~5배 빠름. GIL 존재하나 STM 연구 중. |

- **CPython**: `.py` → `.pyc`(바이트코드) → CPython 인터프리터로 실행
- **Jython**: `.py` → Java 바이트코드(`.class`) → JVM으로 실행
- **PyPy**: `.py` → RPython 바이트코드 → JIT 컴파일로 기계어 실행

### OS-063. 로더의 역할은 fork(), exec()와 관련이 있나요?

네, 밀접한 관련이 있습니다.

**실행 파일을 메모리에 적재하는 과정**:

1. **`fork()`**: 부모 프로세스를 복제하여 자식 프로세스를 생성합니다. 이 시점에서는 부모와 동일한 코드/데이터를 공유합니다(Copy-on-Write).

2. **`exec()`**: 새로운 실행 파일을 **현재 프로세스 공간에 적재**합니다. 이 때 **로더**가 핵심 역할을 합니다.
   - 기존 메모리 공간(코드, 데이터, 힙, 스택)을 초기화합니다.
   - 새 실행 파일의 코드와 초기화 데이터를 메모리에 매핑합니다.
   - 동적 링커(`ld.so`)를 통해 필요한 공유 라이브러리를 매핑합니다.
   - PC를 새 프로그램의 진입점(`_start` → `main()`)으로 설정합니다.

따라서 로더는 `exec()` 시스템 콜 내부에서 동작하며, `fork()+exec()` 패턴이 새 프로세스를 생성하는 표준적인 방법입니다.

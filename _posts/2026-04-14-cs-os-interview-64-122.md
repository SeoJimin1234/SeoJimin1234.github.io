## IPC (Inter-Process Communication)

### OS-064. IPC가 무엇이고, 어떤 종류가 있는지 설명해 주세요.

IPC(Inter-Process Communication)는 **프로세스 간 데이터를 주고받는 메커니즘**입니다. 프로세스는 기본적으로 독립된 주소 공간을 가지므로, OS가 제공하는 IPC 수단을 통해서만 통신할 수 있습니다.

| 종류 | 설명 |
|------|------|
| **파이프(Pipe)** | 단방향 통신. 주로 부모-자식 프로세스 간 사용 |
| **네임드 파이프(Named Pipe/FIFO)** | 파이프의 확장. 비관련 프로세스 간 통신 가능 |
| **메시지 큐(Message Queue)** | 커널이 관리하는 큐를 통해 메시지 송수신 |
| **공유 메모리(Shared Memory)** | 동일한 물리 메모리 영역을 공유. 가장 빠른 IPC |
| **세마포어(Semaphore)** | 동기화를 위한 신호 메커니즘 |
| **소켓(Socket)** | 같은 호스트 또는 다른 호스트 간 네트워크 통신 |

---

### OS-065. Shared Memory가 무엇이며, 사용할 때 유의해야 할 점에 대해 설명해 주세요.

공유 메모리(Shared Memory)는 **두 개 이상의 프로세스가 동일한 물리 메모리 영역을 공유하는 IPC 방식**입니다. 커널을 거치지 않고 메모리에 직접 접근하므로 가장 빠른 IPC 방식입니다.

유의해야 할 점은 **동기화 문제**입니다. 여러 프로세스가 동시에 같은 메모리 영역을 읽고 쓰면 Race Condition이 발생할 수 있습니다. 따라서 세마포어나 뮤텍스 같은 동기화 도구를 함께 사용해 접근을 제어해야 합니다.

---

### OS-066. 메시지 큐는 단방향이라고 할 수 있나요?

아닙니다. 메시지 큐는 **양방향 통신이 가능**합니다. 각 메시지에 타입을 지정할 수 있어, 프로세스 A와 B가 같은 큐를 사용하더라도 타입으로 구분해 각자의 메시지를 선별 수신할 수 있습니다.

파이프(Pipe)가 단방향 통신인 것과 달리, 메시지 큐는 동일한 큐에서 방향 구분 없이 양방향으로 메시지를 주고받을 수 있습니다.

## 스레드 안전성 (Thread Safety)

### OS-067. Thread Safe 하다는 것은 어떤 의미인가요?

Thread Safe란 **멀티스레드 환경에서 여러 스레드가 동시에 같은 코드나 자원에 접근해도 항상 올바른 결과를 보장하는 특성**입니다. Race Condition 없이 데이터의 일관성과 무결성이 유지됩니다.

예를 들어 여러 스레드가 동시에 카운터를 증가시켜도 최종 값이 정확히 반영된다면, 그 카운터 구현은 Thread Safe합니다.

---

### OS-068. Thread Safe를 보장하기 위해 어떤 방법을 사용할 수 있나요?

| 방법 | 설명 |
|------|------|
| **뮤텍스(Mutex)/락(Lock)** | 임계 영역을 한 번에 하나의 스레드만 접근하도록 보호 |
| **세마포어(Semaphore)** | 접근 가능한 스레드 수를 제어 |
| **원자 연산(Atomic Operation)** | CPU 수준에서 불가분적으로 실행되는 연산 |
| **불변 객체(Immutable Object)** | 상태 변경 자체가 없으므로 경쟁 조건 없음 |
| **스레드 로컬 스토리지(TLS)** | 각 스레드만의 독립된 데이터 공간 사용 |

---

### OS-069. Peterson's Algorithm이 무엇이며, 한계점에 대해 설명해 주세요.

피터슨 알고리즘은 **두 프로세스/스레드 간의 상호 배제를 소프트웨어만으로 구현한 알고리즘**입니다.

- `flag[]`: 각 프로세스가 임계 영역에 진입하고 싶음을 표시
- `turn`: 어느 프로세스에게 진입 우선권이 있는지 표시

진입 조건: 상대방이 진입 의사가 없거나(`flag[other] == false`), 내 차례(`turn == me`)일 때만 진입.

**한계점**:
1. **2개의 프로세스에서만 동작** — N개로 확장하기 어려움
2. **Busy Waiting(스핀락)** — 대기 중 CPU를 계속 소비
3. **현대 CPU의 명령어 재정렬(Instruction Reordering)** — 컴파일러나 CPU 최적화로 순서가 바뀌면 올바르게 동작하지 않을 수 있음

---

### OS-070. Race Condition이 무엇인가요?

Race Condition(경쟁 상태)은 **두 개 이상의 스레드/프로세스가 공유 자원에 동시에 접근하면서, 실행 순서에 따라 결과가 달라지는 상황**입니다.

예를 들어 두 스레드가 동시에 `count = count + 1`을 실행할 때, 둘 다 같은 값을 읽고 증가시키면 하나의 증가가 손실됩니다. 이처럼 예측 불가능하고 비결정적인 결과가 Race Condition의 특징입니다.

---

### OS-071. Thread Safe를 구현하기 위해 반드시 락을 사용해야 할까요? 그렇지 않다면, 어떤 다른 방법이 있을까요?

아닙니다. 락 없이도 Thread Safe를 구현하는 방법들이 있습니다.

- **원자 연산(Atomic Operations)**: `CAS(Compare-And-Swap)` 같은 하드웨어 지원 원자 연산으로 락 없이 안전한 업데이트 가능
- **불변 객체(Immutable Object)**: 한 번 생성 후 상태 변경이 없으므로 동기화 불필요
- **스레드 로컬 스토리지(TLS)**: 공유하지 않고 각 스레드만의 독립 공간에서 데이터 관리
- **Lock-Free 자료구조**: CAS 기반으로 구현된 큐, 스택 등 — 데드락 위험이 없고 높은 성능

---

### OS-072. Thread Pool, Monitor, Fork-Join에 대해 설명해 주세요.

**Thread Pool**

미리 일정 수의 스레드를 생성해두고, 작업이 들어오면 유휴 스레드에게 할당합니다. 스레드 생성·소멸 비용을 줄이고 무분별한 스레드 생성으로 인한 자원 고갈을 방지합니다.

**Monitor**

공유 자원과 그 자원에 접근하는 메서드를 하나의 단위로 묶어, 한 번에 하나의 스레드만 접근하도록 보장하는 동기화 구조입니다. Java의 `synchronized` 키워드가 대표적인 모니터 구현입니다.

**Fork-Join**

작업을 재귀적으로 더 작은 단위로 분할(Fork)하고, 각각의 결과를 합산(Join)하는 병렬 처리 패턴입니다. Java의 `ForkJoinPool`이 대표적이며, 분할 정복(Divide and Conquer) 알고리즘과 잘 어울립니다.

---

### OS-073. Thread Pool을 사용한다고 가정하면, 어떤 기준으로 스레드의 수를 결정할 것인가요?

작업의 성격에 따라 다릅니다.

- **CPU 바운드 작업**: 코어 수 + 1 정도. 컨텍스트 스위칭 오버헤드를 줄이는 것이 목적.
- **I/O 바운드 작업**: `코어 수 × (1 + 대기 시간 / 처리 시간)`. I/O 대기 중 다른 스레드가 CPU를 활용할 수 있어 코어 수보다 더 많은 스레드가 효율적.

이론적 공식을 기준으로 삼되, 실제로는 부하 테스트와 프로파일링을 통해 최적값을 결정하는 것이 중요합니다.

---

### OS-074. 어떤 데이터를 정렬하려고 합니다. 어떤 방식의 전략을 사용하는 것이 가장 안전하면서도 좋은 성능을 낼 수 있을까요?

멀티스레드 환경에서는 **안전성(thread-safety)** 과 **성능(병렬화)** 두 가지를 모두 고려해야 합니다.

**안전성 문제**

일반적인 정렬 알고리즘은 스레드 안전하지 않습니다. 여러 스레드가 동일한 배열을 동시에 읽고 쓰면 race condition이 발생합니다. 따라서 아래 두 가지 접근 방식 중 하나를 선택합니다.

- **배열을 분할해 각 스레드가 독립된 구간만 담당** → 공유 자원 접근 자체를 없앰 (권장)
- **공유 자원에 락(Lock) 적용** → 안전하지만 락 경합으로 성능 저하

**성능 전략 — 병렬 Merge Sort (Fork-Join)**

분할 정복 구조의 Merge Sort는 병렬화에 가장 적합합니다.

```
[전체 배열]
     ↓  분할
[좌측 절반]  [우측 절반]
  Thread 1     Thread 2   ← 병렬 실행
     ↓              ↓
  정렬 완료      정렬 완료
     ↓  합병 (Merge)
  [최종 정렬 결과]
```

1. 배열을 스레드 수만큼 균등 분할
2. 각 스레드가 독립적으로 자신의 구간을 정렬 (Insertion Sort 또는 Quick Sort)
3. 메인 스레드가 정렬된 구간들을 순차적으로 Merge

Java의 `Arrays.parallelSort()`가 이 방식(ForkJoinPool 기반 병렬 Merge Sort)을 사용합니다.

**Quick Sort는 왜 병렬화에 불리한가?**

피벗 선택 결과에 따라 분할 크기가 불균등해지면 특정 스레드에만 부하가 몰려 **로드 불균형(load imbalance)** 이 발생합니다. 반면 Merge Sort는 항상 절반씩 분할하므로 스레드 간 작업량이 균등합니다.

**결론**: 멀티스레드 환경에서는 **배열을 독립 구간으로 분할 + 병렬 Merge Sort** 조합이 안전성과 성능 모두를 만족하는 가장 검증된 전략입니다.

## 캐시 메모리 (Cache Memory)

### OS-075. 캐시 메모리 및 메모리 계층성에 대해 설명해 주세요.

메모리 계층(Memory Hierarchy)은 속도·비용·용량의 트레이드오프로 구성됩니다.

```
레지스터 > L1 캐시 > L2 캐시 > L3 캐시 > 메인 메모리(RAM) > SSD > HDD
 (빠름/비쌈/소용량)                          (느림/저렴/대용량)
```

캐시 메모리는 **CPU와 메인 메모리 사이의 속도 차이를 줄이기 위한 고속 소용량 메모리**입니다. 자주 사용되는 데이터를 캐시에 보관해 느린 메인 메모리 접근 빈도를 줄입니다.

---

### OS-076. 캐시 메모리는 어디에 위치해 있나요?

캐시 메모리는 CPU 내부 또는 CPU 칩과 매우 가까운 곳에 위치합니다.

- **L1, L2 캐시**: CPU 코어 내부에 위치. 코어마다 독립적으로 존재.
- **L3 캐시**: 여러 코어가 공유. CPU 칩 내부에 포함.

물리적으로 CPU에 가까울수록 접근 속도가 빠르지만 제조 비용이 높아 용량이 작습니다.

---

### OS-077. L1, L2 캐시에 대해 설명해 주세요.

**L1 캐시**

가장 빠르고 가장 작은 캐시(수십 KB)입니다. 각 코어에 독립적으로 존재하며, 명령어 캐시(L1-I)와 데이터 캐시(L1-D)로 분리된 경우도 있습니다.

**L2 캐시**

L1보다 느리지만 더 큰 캐시(수백 KB ~ 수 MB)입니다. 코어마다 존재하거나 일부 설계에서는 두 코어가 공유하기도 합니다. L1 미스 시 L2를 조회합니다.

L3 캐시는 모든 코어가 공유하는 더 큰 캐시로, LLC(Last Level Cache)라고도 불립니다.

---

### OS-078. 캐시에 올라오는 데이터는 어떻게 관리되나요?

캐시가 가득 찼을 때 어떤 데이터를 교체할지 **캐시 교체 정책(Replacement Policy)**으로 결정합니다.

| 정책 | 설명 |
|------|------|
| **LRU(Least Recently Used)** | 가장 오래 전에 사용된 데이터 교체 |
| **LFU(Least Frequently Used)** | 가장 적게 사용된 데이터 교체 |
| **FIFO** | 가장 먼저 들어온 데이터 교체 |
| **Random** | 무작위 교체 |

실제 하드웨어에서는 완전한 LRU 구현 비용이 높아 Pseudo-LRU(PLRU) 방식을 주로 사용합니다.

---

### OS-079. 캐시간의 동기화는 어떻게 이루어지나요?

멀티코어에서는 각 코어가 독립된 L1/L2 캐시를 가지므로, 동일한 메모리 위치를 서로 다른 캐시에 보관할 경우 일관성 문제가 발생합니다.

이를 해결하기 위해 **캐시 일관성 프로토콜(Cache Coherence Protocol)**을 사용합니다.

**MESI 프로토콜**이 대표적입니다. 각 캐시 라인을 4가지 상태로 관리합니다:

| 상태 | 의미 |
|------|------|
| **M(Modified)** | 수정됨, 메인 메모리와 불일치 |
| **E(Exclusive)** | 이 캐시만 보유, 메인 메모리와 일치 |
| **S(Shared)** | 여러 캐시가 공유, 메인 메모리와 일치 |
| **I(Invalid)** | 무효. 다시 읽어야 함 |

한 코어가 데이터를 수정하면 다른 코어의 동일 캐시 라인을 Invalid로 표시하고, 해당 코어가 다시 읽을 때 최신 값을 가져오도록 합니다.

---

### OS-080. 캐시 메모리의 Mapping 방식에 대해 설명해 주세요.

**직접 매핑(Direct Mapped)**

메모리 블록이 특정 하나의 캐시 라인에만 매핑됩니다. 구현이 간단하지만, 같은 캐시 라인에 매핑되는 블록들이 충돌 미스(Conflict Miss)를 일으킬 수 있습니다.

**완전 연관 매핑(Fully Associative)**

메모리 블록이 임의의 캐시 라인 어디에나 저장될 수 있습니다. 충돌이 적지만 모든 라인을 병렬 비교해야 하므로 하드웨어가 복잡하고 비쌉니다.

**집합 연관 매핑(Set Associative)**

위 두 방식의 절충안입니다. 캐시를 여러 집합(Set)으로 나누고, 각 집합 내에서는 자유롭게 저장합니다. N-way set associative라면 각 집합에 N개의 슬롯이 있습니다. 현대 CPU에서 가장 많이 사용합니다.

---

### OS-081. 캐시의 지역성에 대해 설명해 주세요.

지역성(Locality)이란 **프로그램이 특정 데이터에 반복적으로 접근하는 경향**입니다.

- **시간적 지역성(Temporal Locality)**: 한 번 접근한 데이터는 가까운 미래에 다시 접근될 가능성이 높음. 예) 루프 변수, 반복 호출되는 함수.
- **공간적 지역성(Spatial Locality)**: 접근한 데이터 근처의 데이터가 곧 접근될 가능성이 높음. 예) 배열 순차 접근, 명령어 순차 실행.

캐시는 이 지역성을 활용해 자주 사용될 가능성이 높은 데이터를 미리 올려두어 히트율(Hit Rate)을 높입니다.

---

### OS-082. 캐시의 지역성을 기반으로, 이차원 배열을 가로/세로로 탐색했을 때의 성능 차이에 대해 설명해 주세요.

C/C++/Java 등의 언어에서 2차원 배열은 **행 우선(Row-Major Order)**으로 메모리에 저장됩니다.

- **가로 순서(행 우선) 탐색**: 연속된 메모리 주소를 순차적으로 접근 → 공간 지역성 활용 → 캐시 히트율 높음 → **빠름**
- **세로 순서(열 우선) 탐색**: 메모리상에서 멀리 떨어진 주소를 건너뛰며 접근 → 캐시 미스 빈번 → **느림**

대규모 행렬 연산에서 접근 순서만 바꿔도 수십 배 성능 차이가 날 수 있습니다.

---

### OS-083. 캐시의 공간 지역성은 어떻게 구현될 수 있을까요?

캐시는 데이터를 **캐시 라인(Cache Line)** 단위로 저장합니다. 현대 CPU에서 캐시 라인 크기는 보통 **64바이트**입니다.

특정 데이터를 캐시에 올릴 때 해당 데이터만이 아니라 **주변 64바이트를 통째로 가져옵니다**. 따라서 근방의 다른 데이터를 접근할 때 이미 캐시에 존재하게 되어 공간 지역성이 자연스럽게 구현됩니다.

이 덕분에 배열의 첫 번째 원소를 읽으면 인접한 원소들이 같은 캐시 라인에 올라와 이후 접근이 빠릅니다.

---

## 메모리 할당 (Memory Allocation)

### OS-084. 연속할당 방식 세 가지를 설명해주세요. (first-fit, best-fit, worst-fit)

| 방식 | 설명 | 특징 |
|------|------|------|
| **First-Fit (최초 적합)** | 가장 먼저 발견한 충분한 크기의 빈 공간에 할당 | 탐색이 빠름. 앞부분에 외부 단편화 발생 |
| **Best-Fit (최적 적합)** | 크기가 가장 딱 맞는(가장 작은 적합한) 빈 공간에 할당 | 메모리 낭비 최소화. 탐색 시간 길고, 작은 파편 다수 발생 |
| **Worst-Fit (최악 적합)** | 가장 큰 빈 공간에 할당 | 남은 공간이 커서 다른 프로세스도 사용 가능. 큰 공간이 사라지는 문제 |

---

### OS-085. worst-fit은 언제 사용할 수 있을까요?

Worst-Fit은 할당 후 남은 공간이 가장 크기 때문에, **다양한 크기의 요청이 지속적으로 들어오는 환경**에서 유용합니다. 남은 큰 파편이 이후 중간 크기 요청을 수용할 수 있을 때 활용됩니다.

다만 실제로는 큰 연속 공간을 소모해버려 메모리 활용률이 떨어지는 경우가 많아, Best-Fit이나 First-Fit보다 일반적으로 잘 사용되지 않습니다.

---

### OS-086. 성능이 가장 좋은 알고리즘은 무엇일까요?

일반적으로 **First-Fit**과 **Best-Fit**이 Worst-Fit보다 메모리 활용률이 높습니다. 시뮬레이션 결과 **First-Fit이 평균적으로 빠르고 효율적인 경우**가 많습니다.

단, 정답은 없으며 **워크로드(Workload)의 특성**에 따라 달라집니다. 요청 패턴이 균일하다면 Best-Fit이, 다양하다면 First-Fit이 유리할 수 있습니다.

---

### OS-087. Thrashing이란 무엇인가요?

Thrashing(스래싱)은 **프로세스가 실제 작업보다 페이지 교체(Page Swapping)에 더 많은 시간을 소비하는 현상**입니다.

메모리에 너무 많은 프로세스가 올라와 각 프로세스의 워킹 셋(Working Set)을 담기 부족할 때 발생합니다. 페이지 폴트가 연쇄적으로 발생해 CPU가 I/O 대기만 하게 되므로, 오히려 CPU 이용률이 급격히 떨어지는 역설적 상황이 됩니다.

---

### OS-088. Thrashing 발생 시, 어떻게 완화할 수 있을까요?

- **워킹 셋 모델(Working Set Model)**: 각 프로세스의 워킹 셋(최근 접근한 페이지 집합)을 파악하고, 메모리에 다 담지 못할 프로세스를 스왑 아웃(Swap Out)
- **페이지 폴트 빈도 제어(PFF, Page Fault Frequency)**: 페이지 폴트 발생 빈도가 임계값 이상이면 프레임 추가, 이하이면 프레임 회수
- **다중 프로그래밍 수준 감소**: 실행 중인 프로세스 수를 줄여 각 프로세스에 충분한 메모리를 배분

---

## 가상 메모리 (Virtual Memory)

### OS-089. 가상 메모리란 무엇인가요?

가상 메모리(Virtual Memory)는 **물리 메모리(RAM)가 부족해도 더 많은 주소 공간을 프로세스에게 제공하는 기법**입니다.

각 프로세스는 자신만의 연속적인 가상 주소 공간을 가지며, 실제로 필요한 페이지만 물리 메모리에 올리고 나머지는 디스크(스왑 공간)에 저장합니다. 프로세스 입장에서는 메모리가 무한한 것처럼 보입니다.

---

### OS-090. 가상 메모리가 가능한 이유가 무엇일까요?

**참조 지역성(Locality of Reference)** 때문입니다. 프로그램은 특정 시간에 전체 코드와 데이터 중 일부만 집중적으로 사용합니다. 프로그램 전체가 항상 메모리에 있을 필요가 없고, 현재 실행에 필요한 부분(Working Set)만 올려두어도 정상 실행이 가능합니다.

---

### OS-091. Page Fault가 발생했을 때, 어떻게 처리하는지 설명해 주세요.

1. CPU가 가상 주소에 접근 시 페이지 테이블을 확인
2. 해당 페이지가 물리 메모리에 없음 → **페이지 폴트(Page Fault) 발생**
3. OS의 페이지 폴트 핸들러 실행
4. 물리 메모리에 빈 프레임이 있으면 디스크에서 해당 페이지를 읽어 올림
5. 빈 프레임이 없으면 **페이지 교체 알고리즘**으로 희생 페이지를 선택해 교체
6. 페이지 테이블 업데이트 (valid bit 설정)
7. 폴트를 발생시킨 명령어부터 **재실행**

---

### OS-092. 페이지 크기에 대한 Trade-Off를 설명해 주세요.

| | 페이지 크기가 클 때 | 페이지 크기가 작을 때 |
|---|---|---|
| **페이지 테이블 크기** | 항목 수 감소 → 테이블 작아짐 | 항목 수 증가 → 테이블 커짐 |
| **내부 단편화** | 증가 | 감소 |
| **디스크 I/O** | 한 번에 많이 로드 → 효율적 | I/O 횟수 증가 |
| **필요 없는 데이터 로드** | 증가 (낭비 가능) | 감소 (정밀 로드) |

---

### OS-093. 페이지 크기가 커지면, 페이지 폴트가 더 많이 발생한다고 할 수 있나요?

일반적으로 **아닙니다**. 페이지 크기가 크면 한 번에 더 많은 데이터가 메모리에 올라오므로 공간 지역성을 더 잘 활용하여 페이지 폴트 횟수가 오히려 줄어드는 경향이 있습니다.

다만 필요 없는 데이터까지 올라와 유용한 페이지가 메모리에서 밀려날 수 있고, 내부 단편화로 인한 메모리 낭비가 증가합니다.

---

### OS-094. 세그멘테이션 방식을 사용하고 있다면, 가상 메모리를 사용할 수 없을까요?

사용할 수 있습니다. 세그멘테이션에서도 **세그먼트 테이블에 valid bit를 두어** 해당 세그먼트가 메모리에 없으면 세그먼트 폴트를 발생시키고, OS가 디스크에서 로드하는 방식으로 가상 메모리를 구현할 수 있습니다.

현대 OS에서는 세그멘테이션과 페이징을 결합한 방식(Segmentation + Paging)을 사용하는 경우도 있습니다.

---

### OS-095. 세그멘테이션과 페이징의 차이점은 무엇인가요?

| | 페이징 | 세그멘테이션 |
|---|---|---|
| **분할 단위** | 고정 크기(페이지) | 가변 크기(논리적 단위 — 코드/데이터/스택 등) |
| **내부 단편화** | 발생 가능 | 없음 |
| **외부 단편화** | 없음 | 발생 가능 |
| **보호/공유** | 구현 복잡 | 논리적 단위로 용이 |

---

### OS-096. 페이지와 프레임의 차이에 대해 설명해 주세요.

- **페이지(Page)**: **가상 주소 공간**을 고정 크기로 나눈 단위
- **프레임(Frame)**: **물리 메모리(RAM)**를 페이지와 동일한 크기로 나눈 단위

페이지가 물리 메모리에 로드되면 특정 프레임을 차지합니다. 즉, 페이지는 가상 공간의 단위, 프레임은 물리 공간의 단위입니다. 이 매핑 정보를 페이지 테이블이 관리합니다.

---

### OS-097. 내부 단편화와 외부 단편화에 대해 설명해 주세요.

**내부 단편화(Internal Fragmentation)**

할당된 메모리 블록 내부에 사용하지 않는 공간. 예를 들어 4KB 페이지에 3KB 데이터를 올리면 1KB가 낭비됩니다. **페이징**에서 주로 발생합니다.

**외부 단편화(External Fragmentation)**

메모리 전체 여유 공간은 충분하지만, 연속된 공간이 없어 할당하지 못하는 상황. **세그멘테이션**이나 가변 크기 연속 할당에서 발생합니다.

---

### OS-098. 페이지에서 실제 주소를 어떻게 가져올 수 있는지 설명해 주세요.

가상 주소는 `페이지 번호(Page Number) + 페이지 오프셋(Offset)`으로 구성됩니다.

1. 가상 주소에서 **페이지 번호**를 추출
2. **페이지 테이블**에서 페이지 번호 → **프레임 번호(Frame Number)**로 변환
3. 물리 주소 = `프레임 번호 × 페이지 크기 + 오프셋`

예를 들어 페이지 크기가 4KB일 때, 가상 주소의 상위 비트가 페이지 번호, 하위 12비트가 오프셋입니다.

---

### OS-099. 어떤 주소공간이 있을 때, 이 공간이 수정 가능한지 확인할 수 있는 방법이 있나요?

페이지 테이블의 각 항목에는 **보호 비트(Protection Bit)**가 포함되어 있습니다.

- **R(Read)**: 읽기 가능 여부
- **W(Write)**: 쓰기 가능 여부. Read-Only 영역에 쓰기 시도 시 보호 폴트(Protection Fault) 발생
- **X(Execute, NX/XD bit)**: 실행 가능 여부. 데이터 영역의 코드 실행을 방지

Unix 계열에서는 `mprotect()` 시스템 콜로 특정 메모리 영역의 접근 권한을 동적으로 설정하거나 조회할 수 있습니다.

---

### OS-100. 32비트에서, 페이지의 크기가 1KB이라면 페이지 테이블의 최대 크기는 몇 개일까요?

- 32비트 주소 공간: 2³² = 4GB
- 페이지 크기: 1KB = 2¹⁰ bytes
- **페이지 수 = 2³² ÷ 2¹⁰ = 2²² = 4,194,304개**

---

### OS-101. 32비트 운영체제는 RAM을 최대 4G까지 사용할 수 있습니다. 이 이유를 페이징과 연관 지어서 설명해 주세요.

32비트 운영체제에서 주소는 **32비트**로 표현됩니다. 따라서 표현 가능한 서로 다른 주소의 수는 2³² = 4,294,967,296개, 즉 약 **4GB**입니다.

페이징 관점에서 가상 주소는 페이지 번호와 오프셋으로 이루어지는데, 32비트 체계에서는 최대 4GB의 가상 주소 공간밖에 표현할 수 없습니다. 물리 주소도 이 범위 안에서 매핑되므로, 32비트 OS는 4GB 이상의 RAM을 직접 주소 지정할 수 없습니다. (PAE 기술로 4GB 이상 지원이 가능하나 복잡성이 증가합니다.)

---

### OS-102. C/C++ 개발을 하게 되면 Segmentation Fault라는 에러를 접할 수 있는데, 이 에러는 세그멘테이션/페이징과 어떤 관계가 있을까요?

Segmentation Fault는 프로세스가 **접근 권한이 없는 메모리 주소**에 접근했을 때 OS가 발생시키는 신호(SIGSEGV)입니다.

페이징 기반 시스템에서 다음 경우에 페이지 폴트가 발생하고, OS가 이를 처리할 수 없으면 SIGSEGV 시그널로 프로세스를 종료합니다.

- 페이지 테이블에 매핑되지 않은 주소 접근 (NULL 포인터, 해제된 메모리 접근 등)
- 보호 비트에 의해 금지된 접근 (읽기 전용 영역에 쓰기, NX bit가 설정된 영역에서 실행 등)

이름은 세그멘테이션에서 유래했지만, 현대 시스템에서는 **페이징 보호 위반**과 관련이 큽니다.

---

## TLB & MMU

### OS-103. TLB는 무엇인가요?

TLB(Translation Lookaside Buffer)는 **페이지 테이블 조회를 빠르게 하기 위한 하드웨어 캐시**입니다.

가상 주소 → 물리 주소 변환 시 매번 메인 메모리의 페이지 테이블을 참조하면 느리므로, **최근 사용된 페이지 번호 → 프레임 번호 매핑**을 TLB에 저장해두어 빠르게 변환합니다.

---

### OS-104. TLB를 쓰면 왜 빨라지나요?

TLB 히트(TLB Hit) 시 메인 메모리 접근 없이 **하드웨어 레지스터에서 바로 프레임 번호를 얻을 수 있어** 매우 빠릅니다. TLB 미스(TLB Miss) 시에만 메모리의 페이지 테이블을 조회합니다.

프로그램의 지역성 덕분에 TLB 히트율은 일반적으로 **95% 이상**으로 높아, 평균 메모리 접근 시간을 크게 줄여줍니다.

---

### OS-105. MMU가 무엇인가요?

MMU(Memory Management Unit)는 **가상 주소를 물리 주소로 변환하는 하드웨어 장치**입니다.

CPU가 가상 주소를 생성하면 MMU가 이를 물리 주소로 변환하여 실제 메모리에 접근합니다. TLB를 내부에 포함하고 있으며, 페이지 테이블을 참조해 주소 변환과 메모리 보호(Protection Bit 검사)를 함께 담당합니다.

---

### OS-106. TLB와 MMU는 어디에 위치해 있나요?

MMU는 **CPU 칩 내부**에 위치하며, TLB는 MMU 내부에 포함되어 있습니다. CPU 코어마다 독립된 MMU와 TLB가 존재합니다. 이를 통해 메모리 접근 시 주소 변환이 CPU 내부에서 하드웨어 속도로 처리됩니다.

---

### OS-107. 코어가 여러 개라면, TLB는 어떻게 동기화할 수 있을까요?

각 코어는 독립된 TLB를 가지므로, 페이지 테이블이 수정(매핑 변경, 페이지 해제 등)될 때 다른 코어의 TLB에 오래된 정보가 남아 있는 일관성 문제가 발생합니다.

이를 해결하기 위해 **TLB 슈팅다운(TLB Shootdown)**을 사용합니다:

1. 한 코어가 페이지 테이블을 수정
2. 다른 모든 코어에게 **IPI(Inter-Processor Interrupt)**를 전송
3. 각 코어가 인터럽트를 받아 해당 TLB 항목을 **무효화(Invalidate)**

---

### OS-108. TLB 관점에서, Context Switching 발생 시 어떤 변화가 발생하는지 설명해 주세요.

프로세스마다 각자의 가상 주소 공간이 있으므로, Context Switching 시 이전 프로세스의 TLB 항목이 다음 프로세스에게는 유효하지 않습니다.

**해결 방법**:

- **TLB 전체 플러시(Flush)**: 컨텍스트 스위칭 시 TLB를 모두 비움. 다음 프로세스는 TLB가 비어있어 초기에 미스가 많이 발생하는 성능 저하가 있음.
- **ASID(Address Space IDentifier)**: 각 프로세스에 고유 ID를 부여하여 TLB 항목에 ASID를 태깅. 컨텍스트 스위칭 시 TLB를 비우지 않아도 되어 성능상 유리. ARM, RISC-V 등 현대 아키텍처에서 사용.

---

## 동기화 (Synchronization)

### OS-109. 동기화를 구현하기 위한 하드웨어적인 해결 방법에 대해 설명해 주세요.

| 방법 | 설명 |
|------|------|
| **Test-And-Set(TAS)** | 원자적으로 메모리 위치를 읽고 1로 설정. 스핀락 구현에 사용. |
| **Compare-And-Swap(CAS)** | 기대값과 현재값이 같으면 원자적으로 새 값으로 교체. Lock-Free 알고리즘의 핵심. |
| **Fetch-And-Add** | 원자적으로 메모리 값을 읽고 증가. 티켓 락(Ticket Lock) 구현 등에 사용. |
| **메모리 배리어(Memory Barrier/Fence)** | CPU나 컴파일러의 명령어 재정렬을 방지해 올바른 순서를 보장. |

이 하드웨어 기능들을 기반으로 뮤텍스, 세마포어, RWLock 등의 소프트웨어 동기화 도구가 구현됩니다.

---

### OS-110. volatile 키워드는 어떤 의미가 있나요?

`volatile`은 **컴파일러가 해당 변수를 최적화하지 않도록** 지시하는 키워드입니다. 컴파일러는 반복 접근 변수를 레지스터에 캐싱할 수 있는데, `volatile`을 붙이면 매번 **메모리에서 직접 읽도록** 강제합니다.

주로 메모리 매핑 하드웨어 레지스터, ISR(인터럽트 서비스 루틴)에서 변경되는 변수에 사용합니다.

단, `volatile`은 **원자성을 보장하지 않으므로** 멀티스레드 동기화에는 불충분합니다. Java의 `volatile`은 가시성(Visibility)을 보장하지만, 복합 연산(`i++` 등)의 원자성은 보장하지 않습니다.

---

### OS-111. 싱글코어가 아니라 멀티코어라면, 어떻게 동기화가 이뤄질까요?

멀티코어에서는 각 코어가 독립된 캐시를 가지므로, 싱글코어 방식의 소프트웨어적 동기화만으로는 부족합니다.

- **캐시 일관성 프로토콜(MESI 등)**: 하드웨어 수준에서 캐시 간 일관성 유지
- **원자 명령어(Atomic Instructions)**: 버스 잠금(Bus Lock)이나 캐시 잠금(Cache Lock)으로 멀티코어에서도 원자성 보장
- **메모리 배리어(Memory Barrier)**: CPU 간 명령어 순서 일관성 보장

소프트웨어 수준의 뮤텍스, 세마포어 등은 위 하드웨어 기능을 활용해 구현됩니다.

---

## 페이지 교체 알고리즘 (Page Replacement Algorithm)

### OS-112. 페이지 교체 알고리즘에 대해 설명해 주세요.

물리 메모리가 가득 찼을 때 어떤 페이지를 교체할지 결정하는 알고리즘입니다.

| 알고리즘 | 설명 | 특징 |
|---------|------|------|
| **FIFO** | 가장 먼저 올라온 페이지 교체 | 구현 간단. Belady's Anomaly 발생 가능 |
| **OPT(Optimal)** | 앞으로 가장 오랫동안 사용되지 않을 페이지 교체 | 이론적 최적. 미래를 알 수 없어 실제 구현 불가 |
| **LRU** | 가장 오래 전에 사용된 페이지 교체 | 성능 우수. 구현 비용 있음 |
| **LFU** | 가장 적게 사용된 페이지 교체 | 최근성 반영 부족 |
| **Clock(2nd Chance)** | FIFO + 참조 비트 활용한 LRU 근사 | 실제 OS에서 많이 사용 |

---

### OS-113. LRU 알고리즘은 어떤 특성을 이용한 알고리즘이라고 할 수 있을까요?

**시간적 지역성(Temporal Locality)**을 이용합니다. 최근에 사용된 데이터는 가까운 미래에도 다시 사용될 가능성이 높다는 성질을 기반으로, 가장 오래 전에 사용된 페이지는 앞으로도 사용될 가능성이 낮다고 판단해 교체 대상으로 선정합니다.

---

### OS-114. LRU 알고리즘을 구현한다면, 어떻게 구현할 수 있을까요?

**더블 링크드 리스트 + 해시맵** 조합으로 O(1) 시간 복잡도로 구현할 수 있습니다.

- **해시맵**: 키(페이지 번호) → 리스트 노드 포인터. O(1) 접근.
- **더블 링크드 리스트**: 사용 순서 유지. 맨 앞이 가장 최근, 맨 뒤가 가장 오래된 페이지.

동작:
1. 페이지 히트 시 → 해당 노드를 리스트 맨 앞으로 이동
2. 페이지 미스 시 → 리스트 맨 뒤 노드를 제거하고 새 노드를 맨 앞에 추가

하드웨어에서는 모든 접근에 타임스탬프를 달거나 스택으로 근사 구현합니다.

---

### OS-115. LRU 알고리즘의 단점을 설명해 주세요. 이를 해결할 수 있는 대안에 대해서도 설명해 주세요.

**단점**:
- 과거 사용 이력이 미래 사용을 항상 잘 예측하지는 못함
- **순환 접근(Cyclic Scan)** 패턴에서 성능 저하 — 캐시보다 큰 크기의 배열을 순환 접근하면 모든 페이지가 교체됨
- 하드웨어 구현 비용이 높음

**대안**:
- **LFU(Least Frequently Used)**: 빈도 기반. 최근성을 반영하지 못하는 단점.
- **ARC(Adaptive Replacement Cache)**: LRU + LFU 결합. 최근 접근과 빈도를 모두 고려해 적응적으로 조절.
- **Clock(2nd Chance)**: 참조 비트를 활용한 LRU 근사. 실용적이고 구현이 간단.
- **CLOCK-Pro**: Clock의 개선 버전으로 Hot/Cold 구분을 추가.

## 파일 시스템 (File System)

### OS-116. File Descriptor와 File System에 대해 설명해 주세요.

**File Descriptor(FD)**

열린 파일을 식별하는 **정수 핸들**입니다. 프로세스마다 FD 테이블이 있고, 0(stdin), 1(stdout), 2(stderr)는 기본 할당됩니다. `open()` 시스템 콜 호출 시 FD를 반환합니다.

**File System**

디스크에 파일을 저장하고 관리하는 **논리적 구조**입니다. 디렉토리 구조, 파일 메타데이터(이름, 크기, 권한, 위치 등)를 관리합니다.

| 파일 시스템 | 주요 사용처 |
|------------|------------|
| EXT4 | Linux |
| NTFS | Windows |
| APFS | macOS |
| FAT32 | USB, 임베디드 |

---

### OS-117. I-Node가 무엇인가요?

I-Node(Index Node)는 Unix/Linux 파일 시스템에서 **파일의 메타데이터를 저장하는 자료구조**입니다.

I-Node에 저장되는 정보:
- 파일 크기, 소유자(UID/GID), 접근 권한(rwx)
- 생성/수정/접근 타임스탬프
- 실제 데이터 블록의 위치를 가리키는 포인터들

**주의**: 파일 이름은 I-Node에 포함되지 않습니다. 디렉토리가 `파일명 → I-Node 번호` 매핑을 담당합니다.

- **하드 링크**: 같은 I-Node를 가리키는 여러 파일명
- **심볼릭 링크**: 다른 경로를 가리키는 별도의 파일(별도 I-Node 보유)

---

### OS-118. 프로그래밍 언어 상에서 제공하는 파일 관련 함수는 파일을 어떤 방식으로 읽어들이나요?

1. 라이브러리 함수(Python `open()`, Java `BufferedReader` 등)는 내부적으로 `open()`, `read()` 등의 **시스템 콜**을 호출
2. OS는 파일 시스템을 통해 **I-Node를 조회**하여 데이터 블록 위치를 확인
3. 디스크에서 **블록 단위**로 데이터를 읽어 **커널 버퍼(Page Cache)**에 저장
4. 커널 버퍼에서 **사용자 공간 버퍼**로 복사
5. `BufferedReader`처럼 라이브러리 단계의 버퍼링을 추가하면 시스템 콜 호출 횟수를 줄여 성능을 향상시킴

---

## I/O

### OS-119. 동기와 비동기, 블로킹과 논블로킹의 차이에 대해 설명해 주세요.

**동기 vs 비동기** — 결과 처리의 주체

- **동기(Synchronous)**: 작업 완료를 **호출자가 직접 확인** (결과를 기다림)
- **비동기(Asynchronous)**: 작업 완료를 **콜백/이벤트로 통보**받음 (기다리지 않음)

**블로킹 vs 논블로킹** — 제어권 반환 여부

- **블로킹(Blocking)**: 작업이 완료될 때까지 **호출한 스레드가 대기** (제어권 없음)
- **논블로킹(Non-Blocking)**: 시스템 콜이 **즉시 반환**되고 스레드는 계속 실행 가능 (제어권 유지)

| | 블로킹 | 논블로킹 |
|---|---|---|
| **동기** | 결과 기다리며 스레드 대기 | 결과 확인을 위해 직접 폴링 |
| **비동기** | 거의 사용 안 함 | 콜백/이벤트 기반 (Node.js 등) |

---

### OS-120. 그렇다면, 동기이면서 논블로킹이고, 비동기이면서 블로킹인 경우는 의미가 있다고 할 수 있나요?

**동기 + 논블로킹**: 의미 있습니다. 논블로킹 소켓에서 `recv()`를 반복 호출(폴링)하여 결과가 준비됐는지 직접 확인합니다. 스레드는 멈추지 않고 다른 작업을 병행할 수 있지만, 폴링으로 인한 CPU 낭비가 발생합니다.

**비동기 + 블로킹**: 실질적으로 의미가 없습니다. 비동기 작업을 요청했지만 완료를 기다리며 블로킹하면 비동기의 이점이 사라집니다. 잘못 구현된 경우에 우연히 발생하는 패턴입니다.

---

### OS-121. I/O 멀티플렉싱에 대해 설명해 주세요.

I/O 멀티플렉싱은 **하나의 스레드로 여러 I/O 이벤트를 동시에 감시하는 기법**입니다.

| 메커니즘 | 설명 |
|---------|------|
| **select()** | FD 집합을 감시. FD 수 제한(보통 1024). |
| **poll()** | select와 유사하지만 FD 수 제한 없음. |
| **epoll()** (Linux) | 등록된 FD 중 이벤트 발생한 것만 반환. O(1) 성능. 대규모 연결에 최적. |
| **kqueue** (BSD/macOS) | epoll과 유사한 고성능 이벤트 알림. |
| **IOCP** (Windows) | 완료 포트 기반 비동기 I/O. |

Node.js, Nginx 등의 고성능 서버가 이 방식으로 **적은 스레드로 수만 개의 연결을 처리**합니다.

---

### OS-122. 논블로킹 I/O를 수행한다고 하면, 그 결과를 어떻게 수신할 수 있나요?

| 방법 | 설명 |
|------|------|
| **폴링(Polling)** | 주기적으로 완료 여부를 직접 확인. 간단하지만 CPU 낭비. |
| **I/O 멀티플렉싱(epoll/kqueue 등)** | 이벤트 발생 시 알림. 효율적. |
| **시그널(Signal)** | SIGIO 등의 시그널로 완료 통보. |
| **콜백(Callback)/이벤트 루프** | I/O 완료 시 등록된 콜백 호출. Node.js 방식. |
| **AIO(Asynchronous I/O)** | `aio_read()` 등으로 요청 후 완료 시 시그널 또는 콜백으로 통보. |

논블로킹 I/O와 이벤트 루프를 결합한 방식이 현대 고성능 서버(Node.js, Nginx)의 핵심 아키텍처입니다.

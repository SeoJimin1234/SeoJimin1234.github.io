> CS 기초 스터디 — RAID(Redundant Array of Independent Disks) 레벨별 구조·성능·용도 정리본입니다.

---

## 1. RAID란?

**RAID(Redundant Array of Independent Disks)**는 여러 개의 물리 디스크를 하나의 논리 디스크처럼 묶어 운용하는 기술입니다. 1987년 UC 버클리의 논문 "A Case for Redundant Arrays of Inexpensive Disks"에서 처음 제안되었습니다.

RAID를 사용하는 주된 목적은 두 가지입니다.

| 목적 | 설명 |
|------|------|
| **성능 향상** | 여러 디스크에 데이터를 분산 저장(스트라이핑)해 I/O 병렬 처리 |
| **가용성/신뢰성 향상** | 중복 데이터(미러링) 또는 패리티(Parity)를 저장해 디스크 장애 시 복구 |

### 핵심 용어

| 용어 | 설명 |
|------|------|
| **스트라이핑(Striping)** | 데이터를 청크(chunk) 단위로 여러 디스크에 분산 저장하는 방식 |
| **미러링(Mirroring)** | 동일한 데이터를 두 개 이상의 디스크에 복제 저장하는 방식 |
| **패리티(Parity)** | XOR 연산으로 생성한 오류 복구용 데이터 |
| **스트라이프 폭(Stripe Width)** | 하나의 스트라이프에 포함된 디스크 수 |

## 2. RAID 레벨 한눈에 보기

| 레벨 | 별명 | 최소 디스크 | 장애 허용 | 실 용량 | 주요 특징 |
|------|------|:-----------:|:---------:|:-------:|-----------|
| RAID 0 | Striping | 2 | 0개 | 100% | 최고 성능, 무결성 없음 |
| RAID 1 | Mirroring | 2 | 1개 | 50% | 최고 가용성, 비용 高 |
| RAID 4 | Dedicated Parity | 3 | 1개 | (N-1)/N | 전용 패리티 디스크 병목 |
| RAID 5 | Distributed Parity | 3 | 1개 | (N-1)/N | 분산 패리티, 범용 |
| RAID 6 | Double Parity | 4 | 2개 | (N-2)/N | 이중 패리티, 높은 안정성 |
| RAID 0+1 | Mirror of Stripes | 4 | 1 스트라이프 세트 | 50% | 스트라이프 후 미러 |
| RAID 1+0 | Stripe of Mirrors | 4 | 각 미러 1개씩 | 50% | 미러 후 스트라이프 |

## 3. RAID 0 — Striping (스트라이핑)

![RAID 0 구조](/assets/images/raid/raid0.png)

### 구조

데이터를 청크 단위로 나눠 **모든 디스크에 균등하게 분산** 저장합니다. 중복 데이터나 패리티가 전혀 없습니다.

- 디스크 2개 기준: `[A1, A3, A5...]` / `[A2, A4, A6...]`
- 실제 사용 가능 용량 = 디스크 개수 × 디스크 용량 (100%)

### 특징

- **장점** : 읽기·쓰기 모두 최고 성능. 모든 용량 100% 활용
- **단점** : 디스크 1개라도 고장 시 **전체 데이터 유실** (장애 허용 = 0)
- **용도** : 영상 편집, 임시 캐시, 성능이 최우선이고 데이터 손실을 감수할 수 있는 환경

### 성능

| 항목 | 성능 | 이유 |
|------|:----:|------|
| 읽기 | ★★★★★ | 모든 디스크에서 병렬 읽기 |
| 쓰기 | ★★★★★ | 모든 디스크에 병렬 쓰기 |
| 수정 | ★★★★★ | 패리티 계산 없음, 해당 청크만 덮어씀 |

---

## 4. RAID 1 — Mirroring (미러링)

![RAID 1 구조](/assets/images/raid/raid1.png)

### 구조

동일한 데이터를 **2개의 디스크에 완전히 복제**합니다. 두 디스크는 항상 동일한 내용을 유지합니다.

- 디스크 2개 기준: `[A1, A2, A3...]` / `[A1, A2, A3...]` (완전 동일)
- 실제 사용 가능 용량 = 전체 용량의 50%

### 특징

- **장점** : 디스크 1개 고장 시 나머지로 즉시 운용 가능. 읽기 성능 향상 가능
- **단점** : 용량 효율 50%로 비용이 가장 비쌈
- **용도** : OS 드라이브, 금융·의료 데이터처럼 가용성이 절대적으로 중요한 환경

### 성능

| 항목 | 성능 | 이유 |
|------|:----:|------|
| 읽기 | ★★★★☆ | 두 디스크에서 번갈아 읽기 가능 (부하 분산) |
| 쓰기 | ★★★☆☆ | 두 디스크에 동시 쓰기 필요 → 단일 디스크와 유사 |
| 수정 | ★★★☆☆ | 두 디스크 모두 업데이트 필요 |

---

## 5. RAID 4 — Dedicated Parity (전용 패리티)

![RAID 4 구조](/assets/images/raid/raid4.png)

### 구조

데이터를 N-1개 디스크에 스트라이핑하고, **패리티를 별도의 전용 디스크 1개**에만 저장합니다.

- 디스크 4개 기준 (데이터 3 + 패리티 1):
  - Disk 0: `A1, B1, C1`
  - Disk 1: `A2, B2, C2`
  - Disk 2: `A3, B3, C3`
  - Disk P: `P(A), P(B), P(C)` ← 전용 패리티 디스크

### 특징

- **장점** : 패리티로 디스크 1개 장애 복구 가능. 데이터 디스크 읽기 병렬화
- **단점** : **패리티 디스크에 I/O가 집중** → 병목 발생. 랜덤 쓰기 성능 심각하게 저하
- **용도** : 현재는 거의 사용하지 않음. RAID 5가 이를 개선하여 대체

### 성능

| 항목 | 성능 | 이유 |
|------|:----:|------|
| 읽기 | ★★★★☆ | 데이터 디스크 병렬 읽기 |
| 쓰기 | ★★☆☆☆ | 패리티 디스크 병목 — 모든 쓰기가 패리티 디스크를 거침 |
| 수정 | ★★☆☆☆ | Read-Modify-Write 필요 + 패리티 디스크 병목 |

---

## 6. RAID 5 — Distributed Parity (분산 패리티)

![RAID 5 구조](/assets/images/raid/raid5.png)

### 구조

데이터와 패리티를 **모든 디스크에 분산** 저장합니다. RAID 4의 패리티 병목 문제를 해결한 버전입니다.

- 디스크 4개 기준 예시:

| 스트라이프 | Disk 0 | Disk 1 | Disk 2 | Disk 3 |
|:----------:|:------:|:------:|:------:|:------:|
| 0 | A1 | A2 | A3 | **P(A)** |
| 1 | B1 | B2 | **P(B)** | B3 |
| 2 | C1 | **P(C)** | C2 | C3 |
| 3 | **P(D)** | D1 | D2 | D3 |

### 특징

- **장점** : 패리티 분산으로 병목 해소. 디스크 1개 장애 복구 가능. 용량 효율 (N-1)/N
- **단점** : 쓰기 시 Read-Modify-Write(RMW) 필요 → 랜덤 쓰기 성능 저하
- **용도** : 현재 가장 널리 쓰이는 RAID. NAS, 서버 스토리지 표준

### Read-Modify-Write 과정

```
1. [Read]   해당 스트라이프의 기존 데이터 블록 읽기
2. [Read]   기존 패리티 블록 읽기
3. [Modify] 새 패리티 = 기존 패리티 XOR 기존 데이터 XOR 새 데이터
4. [Write]  새 데이터 블록 쓰기
5. [Write]  새 패리티 블록 쓰기
→ 쓰기 1번에 디스크 I/O 4번 발생 (Write Penalty)
```

### 성능

| 항목 | 성능 | 이유 |
|------|:----:|------|
| 읽기 | ★★★★☆ | 모든 디스크에서 병렬 읽기 (RAID 0과 유사) |
| 쓰기 | ★★★☆☆ | RMW 패널티로 RAID 0 대비 절반 수준 |
| 수정 | ★★★☆☆ | RMW 패널티 동일하게 적용 |

## 7. RAID 6 — Double Parity (이중 패리티)

![RAID 6 구조](/assets/images/raid/raid6.png)

### 구조

RAID 5를 확장해 **패리티 블록을 2개(P, Q)** 저장합니다. P는 XOR 패리티, Q는 갈루아 필드(Galois Field) 연산 기반의 추가 패리티입니다.

- 디스크 5개 기준: 데이터 3 + 패리티 2
- 실제 사용 가능 용량 = (N-2)/N

### 특징

- **장점** : **동시에 디스크 2개 고장**이 나도 데이터 복구 가능. 대용량 어레이에서 안정성 극대화
- **단점** : 쓰기 시 패리티 2개 계산 필요 → RAID 5보다 쓰기 성능 더 낮음. 디스크 최소 4개 필요
- **용도** : 대용량 NAS, 엔터프라이즈 스토리지, 디스크 수가 많아 동시 장애 가능성이 있는 환경

### 성능

| 항목 | 성능 | 이유 |
|------|:----:|------|
| 읽기 | ★★★★☆ | 데이터 디스크 병렬 읽기 |
| 쓰기 | ★★☆☆☆ | 패리티 2개 계산·저장 → RAID 5보다 Write Penalty 더 큼 |
| 수정 | ★★☆☆☆ | P, Q 두 패리티 모두 RMW 수행 |

## 8. RAID 0+1 — Mirror of Stripes

![RAID 0+1 구조](/assets/images/raid/raid0+1.png)

### 구조

먼저 **RAID 0(스트라이핑)으로 디스크 세트를 구성**한 뒤, 해당 스트라이프 세트 전체를 **RAID 1(미러링)**합니다.

```
[Stripe Set A]  Disk0 + Disk1
        ↕ 미러링
[Stripe Set B]  Disk2 + Disk3
```

- 디스크 4개 기준: Stripe Set A와 B는 항상 동일한 데이터

### 특징

- **장점** : 스트라이핑의 높은 성능 + 미러링의 가용성
- **단점** : **한 스트라이프 세트 내 디스크 1개 고장 시 해당 세트 전체가 무효화** → 다른 세트의 디스크 추가 고장에 취약
- **장애 허용** : 한 쪽 스트라이프 세트 전체 고장까지 허용 (단, 반대 세트 내 동일 포지션 고장 시 복구 불가)

### 성능

| 항목 | 성능 | 이유 |
|------|:----:|------|
| 읽기 | ★★★★★ | 4개 디스크 모두 활용 가능 |
| 쓰기 | ★★★★☆ | 스트라이핑 병렬 쓰기, 단 두 세트 동시 기록 |
| 수정 | ★★★★☆ | 패리티 없음, 두 세트 동시 업데이트 |

## 9. RAID 1+0 — Stripe of Mirrors (= RAID 10)

![RAID 1+0 구조](/assets/images/raid/raid1+0.png)

### 구조

먼저 **RAID 1(미러링) 쌍을 구성**한 뒤, 미러 쌍들을 **RAID 0(스트라이핑)**으로 묶습니다.

```
Mirror Pair 1: Disk0 ↔ Disk1
Mirror Pair 2: Disk2 ↔ Disk3
        ↓ 스트라이핑
[Logical Volume: Pair1 + Pair2]
```

### RAID 0+1 vs RAID 1+0 비교

| 항목 | RAID 0+1 | RAID 1+0 |
|------|----------|----------|
| 구성 순서 | 스트라이프 후 미러 | 미러 후 스트라이프 |
| 장애 후 취약성 | 1개 고장 시 세트 전체 무효화 → 반대편 동일 고장 위험 | 1개 고장 시 해당 미러만 단독 운용 → 더 많은 고장 허용 |
| 재구축(Rebuild) | 세트 전체 재구축 | 해당 디스크 1개만 재구축 |
| **실무 선호도** | 낮음 | **높음** |

### 특징

- **장점** : 각 미러 쌍에서 1개씩 총 N/2개 고장까지 허용 가능. 재구축 속도 빠름
- **단점** : 용량 효율 50%, 최소 4개 디스크 필요
- **용도** : 고성능 + 고가용성이 모두 필요한 DB 서버, 가상화 호스트

### 성능

| 항목 | 성능 | 이유 |
|------|:----:|------|
| 읽기 | ★★★★★ | 4개 디스크 병렬 읽기 |
| 쓰기 | ★★★★☆ | 스트라이핑 병렬 쓰기, 각 미러 쌍 동시 기록 |
| 수정 | ★★★★☆ | 패리티 없음, 미러 쌍 동시 업데이트 |

## 11. 상황별 RAID 선택 가이드

| 상황 | 추천 RAID | 이유 |
|------|-----------|------|
| 최고 성능, 데이터 손실 무관 | **RAID 0** | 100% 용량, 최고 I/O |
| 중요 데이터, 비용 무관 | **RAID 1 / RAID 1+0** | 최고 가용성 |
| 범용 서버, 비용·성능 균형 | **RAID 5** | 가장 보편적 선택 |
| 대용량 + 2중 장애 방어 | **RAID 6** | 디스크 수 많을수록 유리 |
| DB 서버, 고성능 + 고가용성 | **RAID 1+0** | 빠른 재구축, 우수한 장애 허용 |
| RAID 0+1 vs 1+0 선택 | **RAID 1+0** | 장애 후 재구축 범위가 훨씬 작음 |

---

## 12. 정리

RAID는 **성능(Performance)**, **가용성(Availability)**, **용량 효율(Capacity Efficiency)** 세 가지를 트레이드오프하는 기술입니다.

```
성능 최우선          → RAID 0
가용성 최우선         → RAID 1
균형(범용)           → RAID 5
최고 안정성(대용량)   → RAID 6
성능 + 가용성 모두    → RAID 1+0 (RAID 10)
```

실무에서는 RAID 5와 RAID 1+0(10)이 가장 많이 사용되며, 특히 데이터베이스 서버에서는 랜덤 쓰기 성능과 빠른 재구축이 중요하기 때문에 **RAID 1+0**을 선호합니다.

---

> 참고: RAID는 **백업을 대체하지 않습니다.** RAID는 디스크 장애에 대한 가용성을 높이는 기술이지, 데이터 백업 솔루션이 아닙니다. 랜섬웨어, 논리적 오류, 자연재해 등에는 별도 백업 전략이 반드시 필요합니다.

## IP 주소

### NET-059. IP 주소는 무엇이며, 어떤 기능을 하고 있나요?

IP 주소는 인터넷에서 각 장치를 식별하는 **논리적 주소**입니다.

- **계층**: 네트워크 계층(L3)에서 사용
- **호스트 식별**: 전 세계에서 고유한 주소로 장치를 구별
- **경로 지정(라우팅)**: 라우터가 목적지 IP를 보고 패킷을 올바른 경로로 전달
- **구조**: 네트워크 부분(어느 네트워크) + 호스트 부분(해당 네트워크의 어느 장치)

IPv4는 32비트(예: `192.168.1.1`), IPv6는 128비트(예: `2001:db8::1`)로 표현됩니다.

---

### NET-060. IPv6는 IPv4의 주소 고갈 문제를 해결하기 위해 만들어졌지만, 아직도 수많은 기기가 IPv4를 사용하고 있습니다. 고갈 문제를 어떻게 해결할 수 있을까요?

단기적으로는 IPv4 절약 기술로, 장기적으로는 IPv6 전환으로 해결합니다.

- **NAT(Network Address Translation)**: 사설 IP를 공인 IP로 변환. 하나의 공인 IP로 수십~수천 대의 기기가 인터넷 접속 → 가장 널리 사용되는 임시 해결책
- **CIDR(Classless Inter-Domain Routing)**: 클래스 없는 IP 할당으로 주소 낭비 감소
- **DHCP**: 동적 IP 할당으로 사용 중인 기기에만 IP 부여
- **IPv6 점진적 전환**: 듀얼 스택, 터널링 등으로 IPv4와 공존하며 전환 중

---

### NET-061. IPv4와 IPv6의 차이에 대해 설명해 주세요.

| 구분 | IPv4 | IPv6 |
|------|------|------|
| 주소 크기 | 32비트 (약 43억 개) | 128비트 (사실상 무한) |
| 표기 | `192.168.1.1` | `2001:db8::1` |
| 헤더 크기 | 가변 (20~60바이트) | 고정 (40바이트) |
| Checksum | 헤더에 포함 | 없음 (상위 계층 담당) |
| NAT | 필수 | 불필요 (주소 충분) |
| 보안 | IPSec 선택 | IPSec 기본 내장 |
| 브로드캐스트 | 있음 | 없음 (멀티캐스트·애니캐스트로 대체) |
| 자동 설정 | DHCP | SLAAC (주소 자동 설정) |
| 단편화 | 라우터에서 가능 | 출발지에서만 가능 |

---

### NET-062. 수많은 사람들이 유동 IP를 사용하고 있지만, 수많은 공유기에서는 고정 주소를 제공하는 기능이 이미 존재합니다. 어떻게 가능한 걸까요?

**DHCP 고정 할당(Static DHCP / DHCP Reservation)**을 통해 가능합니다.

- DHCP는 일반적으로 IP를 동적으로 할당하지만, **특정 MAC 주소에 항상 같은 IP를 할당**하도록 예약 설정 가능
- 공유기 설정에서 특정 기기의 MAC 주소와 IP를 매핑하면, DHCP가 매번 동일한 사설 IP를 부여
- 외부(인터넷)에서는 여전히 유동 공인 IP이지만, 내부 사설 네트워크에서 해당 기기는 고정 IP처럼 동작

---

### NET-063. IPv4를 사용하는 장비와 IPv6를 사용하는 같은 네트워크 내에서 통신이 가능한가요? 가능하다면 어떤 방법을 사용하나요?

가능합니다. 세 가지 전환 메커니즘이 있습니다.

- **듀얼 스택(Dual Stack)**: 장치가 IPv4와 IPv6 주소를 **모두 보유**. 상대방 프로토콜에 맞춰 자동 선택. 가장 이상적이나 두 프로토콜 모두 관리 필요
- **터널링(Tunneling)**: IPv6 패킷을 IPv4 패킷으로 **캡슐화**하여 IPv4 인프라를 통과 (6in4, Teredo, ISATAP)
- **NAT64/DNS64**: IPv6 전용 클라이언트가 IPv4 전용 서버와 통신할 때, 게이트웨이가 주소와 패킷을 변환

---

### NET-064. IP가 송신자와 수신자를 정확하게 전송되는 것을 보장해 주나요?

아니요. IP는 **Best-Effort(최선형 전달)** 방식으로, 신뢰성을 보장하지 않습니다.

- 패킷 손실, 중복 전달, 순서 바뀜, 지연이 발생할 수 있음
- IP 자체에는 오류 복구 메커니즘이 없음
- **신뢰성이 필요하면 상위 계층의 TCP가 담당**
- IP는 단지 최선을 다해 목적지로 전달하려 할 뿐입니다.

IP를 설계할 때 신뢰성을 네트워크 레벨이 아닌 엔드포인트(Transport Layer)에서 처리하도록 한 것은 인터넷의 핵심 설계 철학(End-to-End Principle)입니다.

---

### NET-065. IPv4에서 수행하는 Checksum과 TCP에서 수행하는 Checksum은 어떤 차이가 있나요?

| 구분 | IPv4 Checksum | TCP Checksum |
|------|--------------|-------------|
| 대상 | IP **헤더**만 검증 | TCP 헤더 + **데이터(페이로드)** + 의사 헤더 |
| 목적 | 헤더 손상 여부 확인 | 전체 세그먼트 무결성 확인 |
| 재계산 | 라우터를 거칠 때마다 필요 (TTL 변경) | 양 끝단에서만 |

- IPv4 Checksum은 데이터(페이로드)를 검증하지 않음
- IPv6는 헤더 Checksum을 없애고 TCP/UDP Checksum에 위임
- TCP Checksum이 더 포괄적인 검증을 수행

---

### NET-066. TTL(Hop Limit)이란 무엇인가요?

TTL(Time To Live)은 **IP 패킷이 네트워크에서 거칠 수 있는 최대 라우터(Hop) 수를 제한하는 값**입니다.

- 라우터를 거칠 때마다 TTL 값 1씩 감소
- TTL이 0이 되면 라우터가 패킷을 폐기하고 **ICMP Time Exceeded** 메시지를 출발지에 전송
- **목적**: 라우팅 루프 방지 (잘못 설정된 경로로 패킷이 무한 순환하는 것 방지)
- 기본값: Windows 128, Linux/macOS 64
- IPv6에서는 **Hop Limit**으로 명칭 변경
- `traceroute`는 TTL을 1씩 증가시키며 각 홉의 라우터 IP를 확인하는 도구

---

### NET-067. IP 주소와 MAC 주소의 차이에 대해 설명해 주세요.

| 구분 | IP 주소 | MAC 주소 |
|------|---------|----------|
| 계층 | 네트워크 계층 (L3) | 데이터링크 계층 (L2) |
| 변경 | 변경 가능 (논리적 주소) | 고정 (물리적, 하드웨어 번인) |
| 크기 | IPv4: 32비트 / IPv6: 128비트 | 48비트 |
| 범위 | 전 세계 (전역 라우팅) | 같은 네트워크 내 (로컬) |
| 역할 | 목적지를 찾아가는 경로 (라우팅) | 같은 네트워크 내 정확한 장치로 전달 |
| 예시 | 192.168.1.1 | 00:1A:2B:3C:4D:5E |

비유: IP 주소는 **우편 주소**, MAC 주소는 **수령인 이름**. IP로 건물을 찾고, MAC으로 정확한 사람에게 전달.

## DHCP, DNS

### NET-068. DHCP가 무엇인지 설명해 주세요.

DHCP(Dynamic Host Configuration Protocol)는 **네트워크에 연결된 장치에 IP 주소와 네트워크 설정을 자동으로 할당**하는 프로토콜입니다.

- 수동으로 IP를 설정하지 않아도 됨 → 관리 편의성
- IP 충돌 방지
- 사용하지 않는 IP 회수 후 재할당 → IP 효율적 관리

---

### NET-069. DHCP는 몇 계층 프로토콜인가요?

**애플리케이션 계층(Layer 7)** 프로토콜입니다.

- UDP 기반 (서버 포트 67, 클라이언트 포트 68)
- IP 주소를 아직 받기 전에도 동작해야 하므로 **브로드캐스트(255.255.255.255)** 사용
- IP 계층보다 상위에서 동작하지만, IP 할당이 완료되기 전 단계부터 브로드캐스트로 통신

---

### NET-070. DHCP는 어떻게 동작하나요?

**DORA** 4단계로 동작합니다.

| 단계 | 방향 | 내용 |
|------|------|------|
| **Discover** | 클라이언트 → 브로드캐스트 | DHCP 서버 탐색 |
| **Offer** | 서버 → 클라이언트 | IP 주소와 설정 정보 제안 |
| **Request** | 클라이언트 → 브로드캐스트 | 특정 서버의 제안 수락 (여러 서버가 응답한 경우 하나 선택) |
| **Acknowledge** | 서버 → 클라이언트 | 임대 확정 및 최종 설정 정보 전달 |

이후 임대 기간의 50%가 지나면 갱신(Renew) 시도, 87.5%가 지나면 브로드캐스트로 재시도합니다.

---

### NET-071. DHCP에서 UDP를 사용하는 이유가 무엇인가요?

세 가지 이유가 있습니다.

1. **TCP 연결 불가**: 클라이언트는 아직 IP 주소가 없어 TCP 3-way handshake(IP 필요)가 불가능
2. **브로드캐스트 필요**: 서버를 모르는 상태에서 브로드캐스트로 탐색해야 하는데, TCP는 브로드캐스트 지원 안 함
3. **단순한 교환**: DORA 4단계의 단순한 요청/응답 구조에서 TCP의 복잡한 신뢰성 메커니즘이 불필요

---

### NET-072. DHCP에서, IP 주소 말고 추가로 제공해주는 정보가 있나요?

네, 다양한 네트워크 설정 정보를 함께 제공합니다.

- **서브넷 마스크**: 네트워크와 호스트 부분 구분
- **기본 게이트웨이**: 다른 네트워크로 나가는 출입구
- **DNS 서버 주소**: 도메인 이름 해석에 사용
- **임대 기간(Lease Time)**: IP 주소 유효 기간
- NTP 서버 주소, 도메인 이름, WINS 서버 등

---

### NET-073. DHCP의 유효기간은 얼마나 긴가요?

환경에 따라 다릅니다.

| 환경 | 일반적인 임대 기간 |
|------|-----------------|
| 가정용 공유기 | 24시간 |
| 기업 네트워크 | 8시간 |
| 공용 Wi-Fi | 1~2시간 |
| 서버 환경 | 며칠 이상 (또는 무기한) |

임대 기간이 **짧을수록**: IP 관리가 효율적이나 DHCP 트래픽 증가  
임대 기간이 **길수록**: 트래픽 감소, 유동 환경에서 IP 낭비 가능

---

### NET-074. DNS에 대해 설명해 주세요.

DNS(Domain Name System)는 **도메인 이름을 IP 주소로 변환하는 인터넷의 분산 계층적 데이터베이스 시스템**입니다.

- "인터넷의 전화번호부"
- `www.google.com` → `142.250.x.x`로 변환
- 계층 구조: Root DNS → TLD DNS(`.com`, `.org`) → 권한 네임서버 → 실제 IP 반환
- **분산 설계**: 단일 장애점 없음, 전 세계에 수천 개의 DNS 서버가 협력

---

### NET-075. DNS는 몇 계층 프로토콜인가요?

**애플리케이션 계층(Layer 7)** 프로토콜입니다.

- 기본: UDP 포트 53 사용
- 필요 시: TCP 포트 53 사용 (응답이 큰 경우, Zone Transfer 등)
- 보안 확장: DNSSEC, DNS over HTTPS(DoH), DNS over TLS(DoT)

---

### NET-076. UDP와 TCP 중 어떤 것을 사용하나요?

기본은 **UDP**이지만, 상황에 따라 **TCP**를 사용합니다.

| 프로토콜 | 사용 상황 |
|---------|----------|
| **UDP** | 일반 DNS 쿼리 (빠른 요청/응답, 단일 패킷으로 충분한 경우) |
| **TCP** | 응답이 512바이트 초과 (EDNS0 미지원 환경) / Zone Transfer (서버 간 레코드 복제) / DNSSEC 응답이 큰 경우 |

UDP를 우선 사용하는 이유: 연결 수립 없이 빠르게 응답 가능 (DNS 쿼리는 보통 수십 바이트)

---

### NET-077. DNS Recursive Query, Iterative Query가 무엇인가요?

**Recursive Query(재귀 쿼리)**
- 클라이언트가 Recursive Resolver에 요청하면, Resolver가 최종 IP를 찾아 반환
- 클라이언트는 단 한 번만 요청, 복잡한 탐색은 Resolver가 담당
- 클라이언트 → Recursive Resolver (최종 IP 반환)

**Iterative Query(반복 쿼리)**
- Recursive Resolver가 각 DNS 서버에 차례로 질의
- 각 서버는 최종 IP가 아닌 "다음에 물어볼 서버 주소" 반환

```
Resolver → Root DNS → "TLD(.com) 서버 주소 알려줌"
Resolver → TLD DNS  → "권한 네임서버 주소 알려줌"
Resolver → 권한 DNS → "최종 IP 반환"
```

실제로는 클라이언트-Resolver 구간은 재귀, Resolver 이후 구간은 반복 쿼리입니다.

---

### NET-078. DNS 쿼리 과정에서 손실이 발생한다면, 어떻게 처리하나요?

UDP 기반이므로 재전송 로직은 클라이언트(Resolver)가 담당합니다.

- 타임아웃(일반적으로 2~5초) 후 **자동 재전송**
- 설정된 재시도 횟수(보통 2~3회) 후 실패 시 **대체 DNS 서버로 폴백**
- 브라우저나 OS에 DNS 캐시가 있으면 **캐시에서 반환**
- SERVFAIL 응답 수신 시 다른 DNS 서버로 쿼리 재시도
- DNS over TCP로 자동 전환하여 재전송하는 구현도 있음

---

### NET-079. 캐싱된 DNS 쿼리가 잘못 될 수도 있습니다. 이 경우, 어떻게 에러를 보정할 수 있나요?

- **TTL(Time To Live) 만료**: DNS 레코드마다 TTL이 설정되어 있어 만료 후 자동 재조회
- **DNS 캐시 플러시**: 클라이언트에서 캐시 수동 삭제
  - Windows: `ipconfig /flushdns`
  - Linux: `systemd-resolve --flush-caches`
  - macOS: `sudo dscacheutil -flushcache`
- **DNSSEC**: DNS 응답의 디지털 서명으로 **캐시 포이즈닝(DNS Cache Poisoning) 방지**
  - 공격자가 잘못된 IP를 캐시에 삽입하는 것을 서명 검증으로 방어
- **DNS over HTTPS/TLS**: 암호화된 DNS 통신으로 중간자 변조 방지

---

### NET-080. DNS 레코드 타입 중 A, CNAME, AAAA의 차이에 대해서 설명해주세요.

| 레코드 | 의미 | 예시 |
|--------|------|------|
| **A** | 도메인 → **IPv4 주소** | `example.com → 93.184.216.34` |
| **AAAA** | 도메인 → **IPv6 주소** | `example.com → 2606:2800::1` |
| **CNAME** | 도메인 → **다른 도메인(별칭)** | `www.example.com → example.com` |

CNAME 주의사항:
- CNAME의 최종 목적지는 반드시 A 또는 AAAA 레코드여야 함
- **루트 도메인에는 CNAME 사용 불가** (MX, SOA 등과 공존 불가)
- CNAME 체인은 성능에 영향을 줄 수 있음

기타 주요 레코드: `MX`(메일 서버), `NS`(네임서버), `TXT`(텍스트/도메인 검증), `SRV`(서비스 위치)

---

### NET-081. hosts 파일은 어떤 역할을 하나요? DNS와 비교하였을 때 어떤 것이 우선순위가 더 높나요?

**hosts 파일**: 로컬 컴퓨터에 저장된 **도메인-IP 정적 매핑 파일**

| 항목 | 내용 |
|------|------|
| 위치 | Linux/macOS: `/etc/hosts` / Windows: `C:\Windows\System32\drivers\etc\hosts` |
| 형식 | `127.0.0.1 localhost` |
| 조회 | DNS 조회 전에 참조 |

**우선순위**: **hosts 파일 > DNS** (기본값)
- OS의 NSS(Name Service Switch) 설정으로 변경 가능 (`/etc/nsswitch.conf`)

**사용 사례**: 개발 환경 테스트, 특정 도메인 차단(`0.0.0.0 ads.example.com`)

**보안 위협**: 악성코드가 hosts 파일을 수정하여 피싱 사이트로 리다이렉션 (DNS Hijacking의 일종)

## OSI 7계층 ~ 웹 동작

### NET-082. OSI 7계층에 대해 설명해 주세요.

OSI(Open Systems Interconnection) 모델은 네트워크 통신을 7개의 계층으로 표준화한 참조 모델입니다.

| 계층 | 이름 | 역할 | 프로토콜/장비 |
|------|------|------|-------------|
| 7 | Application (응용) | 사용자 인터페이스 | HTTP, FTP, DNS, SMTP |
| 6 | Presentation (표현) | 데이터 형식 변환, 암호화 | SSL/TLS, JPEG, ASCII |
| 5 | Session (세션) | 세션 관리, 동기화 | NetBIOS, RPC |
| 4 | Transport (전송) | 종단 간 통신, 포트 기반 | TCP, UDP |
| 3 | Network (네트워크) | IP 기반 라우팅 | IP, ICMP, 라우터 |
| 2 | Data Link (데이터링크) | 프레임 전송, MAC 주소 | Ethernet, Wi-Fi, 스위치 |
| 1 | Physical (물리) | 비트 전송, 물리 신호 | 케이블, 허브, NIC |

실제 TCP/IP 모델은 4계층(Application / Transport / Internet / Link)으로 단순화됩니다.

---

### NET-083. Transport Layer와, Network Layer의 차이에 대해 설명해 주세요.

| 구분 | Transport Layer (L4) | Network Layer (L3) |
|------|---------------------|-------------------|
| 주소 체계 | **포트 번호** | **IP 주소** |
| 통신 범위 | **종단 간(End-to-End)** — 출발 프로세스에서 목적 프로세스 | **홉 간(Hop-by-Hop)** — 라우터를 거칠 때마다 처리 |
| PDU 단위 | 세그먼트(TCP) / 데이터그램(UDP) | 패킷 |
| 역할 | 프로세스 간 통신, 신뢰성, 흐름 제어 | 경로 결정, 라우팅 |
| 프로토콜 | TCP, UDP | IP, ICMP, ARP |

Network Layer는 "어떤 경로로 갈지"를 결정하고, Transport Layer는 "목적지에서 어느 프로세스로 갈지"를 결정합니다.

---

### NET-084. L3 Switch와 Router의 차이에 대해 설명해 주세요.

| 구분 | L3 Switch | Router |
|------|-----------|--------|
| 라우팅 | 가능 (**하드웨어 ASIC** 기반, 초고속) | 가능 (소프트웨어/하드웨어 조합) |
| 속도 | 매우 빠름 | 상대적으로 느림 |
| WAN 연결 | 제한적 | DSL, 광랜, LTE 등 다양한 WAN 지원 |
| NAT, VPN | 제한적 | 지원 |
| 사용 위치 | **내부 네트워크(LAN)** — VLAN 간 라우팅 | **네트워크 경계** — 인터넷 연결 |
| 비용 | 상대적으로 낮음 | 높음 |

---

### NET-085. 각 Layer는 패킷을 어떻게 명칭하나요? 예를 들어, Transport Layer의 경우 Segment라 부릅니다.

| 계층 | PDU 명칭 |
|------|---------|
| Application / Presentation / Session | **메시지(Message)** 또는 **데이터(Data)** |
| Transport | **세그먼트(Segment)** — TCP / **데이터그램(Datagram)** — UDP |
| Network | **패킷(Packet)** |
| Data Link | **프레임(Frame)** |
| Physical | **비트(Bit)** |

---

### NET-086. 각각의 Header의 Packing Order에 대해 설명해 주세요.

**캡슐화(Encapsulation)**: 송신 측에서 상위 계층→하위 계층으로 헤더를 추가

```
[HTTP Data]
  → Transport: [TCP Header | HTTP Data]          = 세그먼트
  → Network:   [IP Header | TCP Header | Data]   = 패킷
  → Data Link: [Eth Header | IP Header | ... | Eth Trailer] = 프레임
  → Physical:  비트 전송
```

**역캡슐화(Decapsulation)**: 수신 측에서 하위 계층→상위 계층으로 헤더를 제거

```
비트 수신 → Ethernet 헤더 제거 → IP 헤더 제거 → TCP 헤더 제거 → HTTP 데이터 전달
```

각 계층은 자신이 추가한 헤더만 처리하고, 나머지는 상위 계층으로 그대로 전달합니다.

---

### NET-087. ARP에 대해 설명해 주세요.

ARP(Address Resolution Protocol)는 **IP 주소를 MAC 주소로 변환**하는 프로토콜입니다.

같은 네트워크 내에서 데이터링크 계층으로 전달하려면 목적지의 MAC 주소가 필요한데, IP 주소만 알고 있을 때 ARP로 MAC을 알아냅니다.

**동작 과정**:
1. 송신자가 "IP X를 가진 기기의 MAC 주소가 무엇인가요?"를 **브로드캐스트** 전송
2. 해당 IP를 가진 기기가 자신의 MAC 주소를 **유니캐스트**로 응답
3. 송신자가 ARP 테이블(캐시)에 IP-MAC 매핑 저장

**보안 위협**: ARP 스푸핑 — 공격자가 잘못된 ARP 응답으로 다른 기기의 MAC 주소를 자신의 MAC으로 교체 → 트래픽 가로채기(MITM)

---

### NET-088. 라우터 내의 포워딩 과정에 대해 설명해 주세요.

1. 인입 포트에서 **패킷 수신**
2. 패킷의 **목적지 IP 주소 추출**
3. **포워딩 테이블(FIB)**에서 목적지 IP에 매칭되는 항목 검색 (**Longest Prefix Match**)
4. 매칭된 항목의 **출력 포트(인터페이스)**로 패킷 전달
5. **TTL 1 감소**, IP Checksum 재계산
6. 출력 링크의 다음 홉 MAC 주소 결정(ARP 사용) 후 **이더넷 프레임 재구성**
7. 다음 라우터(또는 목적지 호스트)로 전송

---

### NET-089. 라우팅과 포워딩의 차이는 무엇인가요?

| 구분 | 라우팅(Routing) | 포워딩(Forwarding) |
|------|----------------|------------------|
| 역할 | 포워딩 테이블을 **만드는** 과정 | 테이블을 **참조하여 패킷을 전달**하는 과정 |
| 계층 | Control Plane | Data Plane |
| 속도 | 느림 (경로 계산) | 매우 빠름 (테이블 조회) |
| 프로토콜 | OSPF, BGP, RIP | - |
| 처리 | 소프트웨어 (일반적으로) | 하드웨어 ASIC |

비유: **라우팅**은 네비게이션이 경로를 계산하는 것, **포워딩**은 실제로 그 경로를 따라 운전하는 것.

---

### NET-090. 라우팅 알고리즘에 대해 설명해 주세요.

**링크 상태(Link-State) 알고리즘**
- 모든 라우터가 전체 네트워크 토폴로지 정보를 공유
- **Dijkstra 최단 경로 알고리즘**으로 각 목적지까지 최단 경로 계산
- 예: OSPF(내부 라우팅), IS-IS
- 장점: 빠른 수렴 / 단점: 많은 메모리·CPU 필요

**거리 벡터(Distance-Vector) 알고리즘**
- 인접 라우터와 자신의 라우팅 테이블(목적지별 비용)을 주기적으로 교환
- **Bellman-Ford 알고리즘** 기반
- 예: RIP
- 장점: 구현 단순 / 단점: 수렴 속도 느림, Count to Infinity 문제

**경로 벡터(Path-Vector) 알고리즘**
- 거리 벡터에 전체 경로 정보를 추가하여 루프 방지
- 예: **BGP** (AS 간 인터넷 라우팅)

---

### NET-091. 포워딩 테이블의 구조에 대해 설명해 주세요.

| 목적지 네트워크 | 프리픽스 | 다음 홉(Next Hop) | 출력 인터페이스 |
|--------------|---------|-----------------|------------|
| 192.168.1.0 | /24 | 10.0.0.1 | eth0 |
| 10.0.0.0 | /8 | - | eth1 (직접 연결) |
| 0.0.0.0 | /0 | 203.0.113.1 | eth2 (기본 게이트웨이) |

**Longest Prefix Match**: 여러 항목이 목적지 IP에 매칭될 때 **가장 긴 서브넷 프리픽스**를 가진 항목 우선 선택
- 예: 목적지 192.168.1.5에 /8과 /24가 모두 매칭되면 /24 선택

**기본 게이트웨이**: `0.0.0.0/0` — 다른 항목에 매칭되지 않는 모든 트래픽을 보내는 출구

### NET-092. 로드밸런서가 무엇인가요?

로드밸런서(Load Balancer)는 **여러 서버에 들어오는 트래픽을 분산시켜 부하를 균등하게 배분**하는 장치 또는 소프트웨어입니다.

- **특정 서버 과부하 방지**: 요청을 여러 서버에 나눔
- **고가용성(HA)**: 일부 서버 장애 시 자동으로 다른 서버로 전환
- **수평 확장(Scale-out) 지원**: 서버 추가로 처리 능력 확장
- 클라이언트는 로드밸런서 IP 하나만 알면 되며, 실제 서버 구성은 투명(Transparent)

---

### NET-093. L4 로드밸런서와, L7 로드밸런서의 차이에 대해 설명해 주세요.

| 구분 | L4 로드밸런서 | L7 로드밸런서 |
|------|-------------|-------------|
| 동작 계층 | 전송 계층 (TCP/UDP) | 응용 계층 (HTTP/HTTPS) |
| 분산 기준 | IP 주소, 포트 번호 | URL 경로, 헤더, 쿠키, 컨텐츠 |
| 패킷 분석 | 최소 (헤더만 확인) | 전체 요청 내용 분석 |
| 속도 | 빠름 | 상대적으로 느림 |
| 기능 | 단순 분산 | URL 기반 라우팅, SSL 종료, 세션 유지, WAF |
| 예시 | AWS NLB, HAProxy (TCP 모드) | AWS ALB, Nginx, HAProxy (HTTP 모드) |

L7은 `/api` 요청을 API 서버로, `/static` 요청을 파일 서버로 분리하는 등 **컨텐츠 기반 라우팅**이 가능합니다.

---

### NET-094. 로드밸런서 알고리즘에 대해 설명해 주세요.

| 알고리즘 | 설명 |
|---------|------|
| **Round Robin** | 순서대로 서버에 배분. 단순하나 서버 성능 차이 무시 |
| **Weighted Round Robin** | 서버 성능에 따라 가중치 부여하여 배분 |
| **Least Connections** | 현재 활성 연결 수가 가장 적은 서버 선택 |
| **Weighted Least Connections** | 가중치 + 최소 연결 수 조합 |
| **IP Hash** | 클라이언트 IP를 해시하여 특정 서버에 고정 → Sticky Session 효과 |
| **Least Response Time** | 응답 시간이 가장 짧은 서버 선택 |
| **Random** | 무작위 서버 선택 |

---

### NET-095. 로드밸런싱 대상이 되는 장치중 일부 장치가 문제가 생겨 접속이 불가능하다고 가정해 봅시다. 이 경우, 로드밸런서가 해당 장비로 요청을 보내지 않도록 하려면 어떻게 해야 할까요?

**헬스 체크(Health Check)**를 사용합니다.

- 로드밸런서가 주기적으로 백엔드 서버에 **헬스 체크 요청 전송** (HTTP GET `/health`, TCP 연결 시도 등)
- **응답이 없거나 비정상 응답** (HTTP 5xx 등) 시 해당 서버를 풀(Pool)에서 자동 제외
- 서버가 복구되면 자동으로 풀에 재등록
- 주요 설정: 체크 간격(Interval), 타임아웃(Timeout), 실패 임계값(Unhealthy threshold), 성공 임계값(Healthy threshold)

---

### NET-096. 로드밸런서 장치를 사용하지 않고, DNS를 활용해서 유사하게 로드밸런싱을 하는 방법에 대해 설명해 주세요.

**DNS 라운드 로빈(DNS Round Robin)**을 사용합니다.

- 하나의 도메인에 여러 IP 주소를 **A 레코드로 등록**
- DNS 서버가 쿼리마다 IP 목록의 **순서를 순환**하여 다른 IP 반환

```
example.com  A  93.184.1.1
example.com  A  93.184.1.2
example.com  A  93.184.1.3
```

**장점**: 추가 장비 불필요, 지리적 분산 가능(지역별 다른 IP)

**단점**:
- 서버 장애를 감지할 수 없어 **장애 서버 IP가 TTL 동안 계속 반환**됨
- DNS 캐싱으로 인해 균등 분산이 보장되지 않음
- 세션 유지(Sticky Session) 불가

---

### NET-097. 서브넷 마스크와, 게이트웨이에 대해 설명해 주세요.

**서브넷 마스크(Subnet Mask)**
- IP 주소에서 **네트워크 부분과 호스트 부분을 구분**하는 32비트 값
- 1비트 부분 = 네트워크, 0비트 부분 = 호스트
- AND 연산: `IP 주소 & 서브넷 마스크 = 네트워크 주소`
- 예: IP `192.168.1.100` & 마스크 `255.255.255.0` = 네트워크 `192.168.1.0`

**기본 게이트웨이(Default Gateway)**
- 다른 네트워크로 나가는 출입구. 일반적으로 같은 네트워크의 **라우터 IP**
- 목적지가 같은 서브넷이 아니면 게이트웨이로 패킷 전달

---

### NET-098. NAT에 대해 설명해 주세요.

NAT(Network Address Translation)은 **사설 IP 주소를 공인 IP 주소로 변환**하는 기술입니다.

**동작 원리**:
1. 내부 기기(192.168.1.10)가 인터넷에 패킷 전송
2. 라우터(공유기)가 **출발지 IP를 공인 IP로 변환** + 포트 번호로 기기 구분
3. NAT 변환 테이블에 매핑 저장
4. 응답 패킷 수신 시 변환 테이블 참조하여 원래 사설 IP로 복원 전달

**NAT 유형**:
- **SNAT(Source NAT)**: 출발지 IP 변환 (내부 → 외부 통신)
- **DNAT(Destination NAT)**: 목적지 IP 변환 (포트 포워딩)
- **PAT/NAPT**: 포트 번호까지 변환하여 하나의 공인 IP로 다수 기기 지원 (가장 일반적)

---

### NET-099. 서브넷 마스크의 표현 방식에 대해 설명해 주세요.

두 가지 표현 방식이 있습니다.

**점-십진수(Dotted-Decimal)**
- `255.255.255.0`

**CIDR 표기법(Prefix Length)**
- `/24` — 연속된 1비트가 24개임을 의미

| CIDR | 점-십진수 | 호스트 수 |
|------|----------|---------|
| /8 | 255.0.0.0 | 16,777,214 |
| /16 | 255.255.0.0 | 65,534 |
| /24 | 255.255.255.0 | 254 |
| /30 | 255.255.255.252 | 2 |
| /32 | 255.255.255.255 | 1 (단일 호스트) |

---

### NET-100. 그렇다면, 255.0.255.0 같은 꼴의 서브넷 마스크도 가능한가요?

기술적으로 설정은 가능하지만, **표준에서 허용하지 않습니다**.

- 표준 서브넷 마스크는 **1이 연속으로 앞부분에만 존재**하는 형태여야 함
  - 유효: `11111111.11111111.11111111.00000000` (255.255.255.0)
  - 비유효: `11111111.00000000.11111111.00000000` (255.0.255.0)
- 비연속 패턴은 **CIDR 표기법으로 표현 불가**
- 대부분의 라우터, OS, 방화벽이 비연속 마스크를 **지원하지 않거나 거부**
- 결론: 이론상 가능하나 실제로는 사용 불가

### NET-101. www.github.com을 브라우저에 입력하고 엔터를 쳤을 때, 네트워크 상 어떤 일이 일어나는지 최대한 자세하게 설명해 주세요.

**1단계 — URL 파싱 및 캐시 확인**
- 브라우저가 URL을 파싱 (프로토콜 `https`, 도메인 `www.github.com`, 경로 `/`)
- 브라우저 캐시 → OS DNS 캐시 → hosts 파일 순서로 IP 확인

**2단계 — DNS 조회**
- 캐시 미스 시 Recursive Resolver에 쿼리
- Root DNS → `.com` TLD DNS → `github.com` 권한 네임서버 → IP 반환
- 결과 캐시 (TTL만큼)

**3단계 — TCP 연결 수립**
- 획득한 IP(CDN 엣지 또는 로드밸런서)로 **TCP 3-way handshake** (SYN → SYN-ACK → ACK)

**4단계 — TLS 핸드셰이크**
- HTTPS이므로 **TLS 협상** (클라이언트 Hello → 서버 인증서 제공 → 키 교환 → 암호화 통신 시작)

**5단계 — HTTP 요청**
- `GET / HTTP/2` 요청 전송 (`Host`, `User-Agent`, `Accept-Encoding` 등 헤더 포함)

**6단계 — 서버 처리 및 응답**
- CDN 히트 시 엣지에서 바로 응답 / 미스 시 Origin 서버 → WAS → DB 조회
- HTML, CSS, JS 응답

**7단계 — 렌더링**
- DOM 파싱 → CSSOM 구성 → Render Tree → Layout → Paint → Composite

**8단계 — 추가 리소스**
- CSS, JS, 이미지에 대한 추가 HTTP 요청 (HTTP/2 병렬 처리)

**9단계 — 연결 정리**
- Keep-Alive로 연결 유지 또는 4-way handshake로 종료

---

### NET-102. DNS 쿼리를 통해 얻어진 IP는 어디를 가리키고 있나요?

일반적으로 **CDN 엣지 서버** 또는 **로드밸런서**의 IP를 가리킵니다.

- 대형 서비스는 단일 원본 서버가 아닌 **CDN 엣지 서버**로 라우팅
- **Anycast**: 동일한 IP이지만 지리적으로 가장 가까운 서버로 자동 라우팅 (Cloudflare, Google DNS 등)
- CDN 없는 경우: 로드밸런서 IP → 실제 백엔드 서버 군으로 분산
- 따라서 DNS 응답 IP ≠ 실제 애플리케이션 서버 IP인 경우가 대부분

### NET-103. Web Server와 Web Application Server의 차이에 대해 설명해 주세요.

| 구분 | Web Server (WS) | Web Application Server (WAS) |
|------|----------------|------------------------------|
| 역할 | **정적 컨텐츠** 제공 | **동적 컨텐츠** 생성 |
| 처리 | HTML, CSS, JS, 이미지 파일 직접 반환 | 비즈니스 로직 실행, DB 연동 |
| 예시 | Nginx, Apache HTTP Server | Tomcat, Node.js, Django, Spring Boot |

**일반적인 조합**:
```
클라이언트 → [Nginx(WS)] → 정적 파일 직접 반환
                         → [Tomcat(WAS)] → DB 조회 → 동적 HTML 생성
```

정적 파일은 WS가 직접 처리해 WAS 부하를 줄이고, WS가 리버스 프록시로 WAS를 보호합니다.

### NET-104. URL, URI, URN은 어떤 차이가 있나요?

| 구분 | 의미 | 예시 |
|------|------|------|
| **URI** | 자원을 **식별**하는 가장 상위 개념. URL과 URN을 포함 | `https://example.com/page`, `urn:isbn:...` |
| **URL** | 자원의 **위치(Location)** — 어디에 있고 어떻게 접근하는지 | `https://example.com/users/1` |
| **URN** | 자원의 **이름(Name)** — 위치와 무관한 영구적 식별자 | `urn:isbn:978-3-16-148410-0` |

- 모든 URL은 URI이지만, 모든 URI가 URL은 아님
- URN은 위치가 바뀌어도 동일한 이름으로 자원 식별 (책의 ISBN처럼)
- 실제 웹 개발에서는 URL이 주로 사용되며, URI와 URL은 혼용됨

### NET-105. gRPC란 무엇이고, REST API와 비교했을 때 어떤 차이점이 있나요?

gRPC(Google Remote Procedure Call)는 Google이 개발한 **오픈소스 고성능 RPC 프레임워크**입니다.

| 구분 | gRPC | REST API |
|------|------|---------|
| 프로토콜 | HTTP/2 | HTTP/1.1, HTTP/2 |
| 직렬화 | **Protocol Buffers** (바이너리) | JSON (텍스트) |
| 성능 | 빠름, 적은 대역폭 | 상대적으로 느림 |
| 스트리밍 | **양방향 스트리밍** 지원 | 제한적 (SSE, WebSocket 별도) |
| 브라우저 지원 | 제한적 (grpc-web 필요) | 완벽 지원 |
| 코드 생성 | `.proto`로 클라이언트/서버 자동 생성 | 수동 또는 OpenAPI |
| 주요 사용처 | 마이크로서비스 내부 통신 | 공개 API, 브라우저 클라이언트 |

### NET-106. Protocol Buffers의 직렬화 방식과 JSON 대비 장단점을 설명해주세요.

Protocol Buffers(protobuf)는 Google이 개발한 **이진(Binary) 직렬화 포맷**입니다.

**직렬화 방식**:
1. `.proto` 파일에 스키마(메시지 구조) 정의
2. `protoc` 컴파일러로 각 언어의 코드 자동 생성
3. **필드 번호 기반** 이진 인코딩 (필드명 대신 숫자로 식별 → 크기 절감)

**protobuf vs JSON 비교**:

| 구분 | Protocol Buffers | JSON |
|------|-----------------|------|
| 크기 | 3~10배 작음 | 상대적으로 큼 |
| 속도 | 훨씬 빠른 직렬화/역직렬화 | 느림 |
| 가독성 | 없음 (이진 형식) | 사람이 읽을 수 있음 |
| 스키마 | 필수 (`.proto` 파일) | 선택 (유연함) |
| 브라우저 | 직접 읽기 불가 | 기본 지원 |
| 디버깅 | 불편 | 편리 |

### NET-107. gRPC의 스트리밍 방식(Unary, Server Streaming, Client Streaming, Bidirectional)을 설명해주세요.

| 유형 | 설명 | 사용 사례 |
|------|------|----------|
| **Unary** | 클라이언트 1요청 → 서버 1응답 | 일반 API 호출 |
| **Server Streaming** | 클라이언트 1요청 → 서버 **여러 응답** 스트리밍 | 실시간 데이터 피드, 로그 스트리밍, 알림 |
| **Client Streaming** | 클라이언트 **여러 요청** 스트리밍 → 서버 1응답 | 대용량 파일 업로드, IoT 센서 데이터 수집 |
| **Bidirectional Streaming** | 클라이언트·서버 **모두 스트리밍** | 실시간 채팅, 게임, 화상통화 |

모두 HTTP/2의 **스트림 멀티플렉싱**을 활용합니다. `.proto`에서 `stream` 키워드로 선언합니다.

### NET-108. CDN(Content Delivery Network)의 동작 원리와 사용 목적을 설명해주세요.

CDN은 **전 세계에 분산된 엣지 서버 네트워크를 통해 사용자에게 가장 가까운 서버에서 컨텐츠를 제공**하는 시스템입니다.

**동작 원리**:
1. DNS가 사용자의 지리적 위치에 따라 **가장 가까운 엣지 서버 IP 반환**
2. 엣지 서버에 캐시가 있으면 → **Cache Hit**: 바로 응답
3. 캐시 미스 → 원본 서버(Origin)에서 컨텐츠 가져와 **캐시 저장** 후 응답

**사용 목적**:
- **레이턴시 감소**: 지리적으로 가까운 서버에서 응답
- **원본 서버 부하 감소**: 캐시 히트로 Origin 요청 최소화
- **대역폭 비용 절감**
- **DDoS 완화**: 엣지 서버가 공격 트래픽 흡수
- **가용성 향상**: 원본 서버 장애 시 캐시된 컨텐츠 계속 제공

### NET-109. REST API 설계 원칙(RESTful 제약 조건)에 대해 설명해주세요.

Roy Fielding이 제안한 6가지 REST 아키텍처 제약 조건입니다.

1. **클라이언트-서버(Client-Server)**: UI(클라이언트)와 데이터 저장(서버)을 분리. 각각 독립적으로 진화 가능

2. **Stateless**: 서버가 클라이언트 상태를 저장하지 않음. 각 요청이 처리에 필요한 모든 정보를 포함

3. **캐시 가능(Cacheable)**: 응답에 캐시 가능 여부를 명시 (`Cache-Control` 헤더). 불필요한 요청 감소

4. **균일한 인터페이스(Uniform Interface)**:
   - 자원을 URI로 식별
   - 표현(Representation)을 통한 자원 조작
   - 자기 서술적 메시지
   - HATEOAS (응답에 다음 가능한 작업 링크 포함)

5. **계층화 시스템(Layered System)**: 클라이언트는 중간 서버(캐시, 로드밸런서, 보안 레이어) 존재 여부를 알 필요 없음

6. **코드 온 디맨드(Code on Demand)**: 선택적. 서버에서 실행 가능한 코드 전달 (JavaScript 등)

### NET-110. API 버전 관리 전략(URL, Header, Query Parameter)의 장단점을 비교해주세요.

| 방식 | 예시 | 장점 | 단점 |
|------|------|------|------|
| **URL 경로** | `/api/v1/users` | 명확, 브라우저·캐시 친화적, 로그 파악 쉬움, 테스트 편리 | URI가 자원 위치 외의 정보 포함 (REST 순수성 논쟁) |
| **쿼리 파라미터** | `/api/users?version=1` | 선택적 버전 지정 가능 | 필수 파라미터 강제 어려움, 캐싱 복잡, 누락 시 처리 필요 |
| **요청 헤더** | `Accept: application/vnd.api.v1+json` | URI 깔끔, 내용 협상과 일치, REST 원칙에 부합 | 브라우저 직접 테스트 불편, 캐싱 어려움, 가시성 낮음 |

**실무 권장**: URL 경로 방식이 가장 많이 사용됨 (명확성, 개발 편의성, 캐시 호환성)

### NET-111. Rate Limiting의 구현 방식(Token Bucket, Leaky Bucket, Sliding Window)을 설명해주세요.

Rate Limiting은 **API 또는 서비스에 대한 요청 빈도를 제한**하여 남용을 방지하는 기법입니다.

**Token Bucket (토큰 버킷)**
- 일정 속도로 버킷에 토큰이 채워짐. 요청 시 토큰 소비, 토큰이 없으면 요청 거절
- **버스트(Burst) 허용**: 쌓인 토큰을 한꺼번에 사용해 단기 고속 요청 가능
- AWS API Gateway, 대부분의 API Rate Limiter에서 사용

**Leaky Bucket (누수 버킷)**
- 요청이 큐에 쌓이고 **고정 속도**로 처리. 큐가 가득 차면 초과 요청 거절
- 균일한 처리 속도 보장
- 버스트 처리 불가 (모든 요청이 동일 속도로 처리)

**Fixed Window Counter (고정 창 카운터)**
- 고정 시간 창(예: 1분)에 요청 수 카운트. 창 초기화 시 리셋
- 단순하나 **경계 시점 버스트 취약점**: 59초와 61초에 각각 한계치 요청 → 2초 내 2배 요청 가능

**Sliding Window Log (슬라이딩 창 로그)**
- 각 요청의 타임스탬프를 저장. 현재 시점 기준으로 창 내 요청 수 실시간 계산
- 정확하지만 **메모리 소비 많음**

**Sliding Window Counter (슬라이딩 창 카운터)**
- Fixed Window의 현재 창 카운트 + 이전 창 카운트의 가중치 합산
- 정확도와 메모리 효율성의 **균형** → Redis 기반 구현에서 많이 사용

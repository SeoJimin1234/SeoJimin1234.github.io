## HTTP 기초

### NET-001. HTTP에 대해 설명해 주세요.

HTTP(HyperText Transfer Protocol)는 웹에서 클라이언트와 서버 간 데이터를 주고받기 위한 **애플리케이션 계층 프로토콜**입니다.

- TCP/IP 기반으로 동작 (HTTP/3는 UDP/QUIC 기반)
- **요청(Request) / 응답(Response)** 구조
- 기본적으로 **Stateless, Connectionless** 특성 보유
- 기본 포트: HTTP 80, HTTPS 443

HTTP 메시지는 **Start Line + Headers + Body** 구조로 이루어집니다. 요청 메시지의 Start Line에는 메서드, 경로, HTTP 버전이, 응답 메시지에는 상태 코드가 포함됩니다.

---

### NET-002. HTTP의 특성인 Stateless에 대해 설명해 주세요.

Stateless란 **서버가 이전 요청의 상태(클라이언트 정보)를 저장하지 않는 것**을 의미합니다.

- 각 요청은 완전히 독립적으로 처리됨
- 서버는 이전 요청과 현재 요청의 관계를 알지 못함
- **장점**: 서버 확장이 쉬움 — 어느 서버가 요청을 받아도 동일하게 처리 가능
- **단점**: 매 요청마다 필요한 정보(인증 토큰 등)를 함께 전송해야 함

---

### NET-003. Stateless와 Connectionless에 대해 설명해 주세요.

두 개념은 다른 차원의 이야기입니다.

| 개념 | 의미 |
|------|------|
| **Stateless** | 서버가 클라이언트의 **상태(State)**를 기억하지 않음 (세션, 인증 정보 등) |
| **Connectionless** | 요청/응답이 완료되면 **TCP 연결을 끊음** (HTTP/1.0 기준) |

- Stateless는 애플리케이션 레벨의 특성이고, Connectionless는 연결 레벨의 특성입니다.
- HTTP/1.1부터는 keep-alive로 연결을 유지(Persistent Connection)하므로 Connectionless 특성은 사실상 완화되었습니다.

---

### NET-004. 왜 HTTP는 Stateless 구조를 채택하고 있을까요?

**확장성(Scalability)과 단순성** 때문입니다.

- 서버가 상태를 저장하지 않으므로 어느 서버에도 요청을 보낼 수 있어 **로드밸런싱이 용이**
- 서버 메모리 절약
- 서버 장애 시 다른 서버로 즉시 전환 가능
- 서버 구현이 단순해짐

인터넷 규모의 웹 서비스에서 수억 개의 클라이언트 상태를 서버가 직접 관리하는 것은 현실적으로 불가능합니다. Stateless 구조 덕분에 서버를 수평 확장(Scale-out)할 수 있습니다.

---

### NET-005. Connectionless의 논리대로면 성능이 되게 좋지 않을 것으로 보이는데, 해결 방법이 있을까요?

여러 방법으로 해결합니다.

- **Persistent Connection (Keep-Alive)**: HTTP/1.1 기본값. 하나의 TCP 연결로 여러 요청/응답 처리
- **파이프라이닝(Pipelining)**: 응답을 기다리지 않고 여러 요청을 연속 전송 (HTTP/1.1, 실제 지원은 미흡)
- **멀티플렉싱(Multiplexing)**: HTTP/2에서 하나의 TCP 연결로 여러 요청을 **병렬**로 처리
- **연결 풀링(Connection Pooling)**: 브라우저/서버가 미리 여러 연결을 열어 재사용

---

### NET-006. TCP의 keep-alive와 HTTP의 keep-alive의 차이는 무엇인가요?

이름은 같지만 목적과 동작 계층이 다릅니다.

| 구분 | TCP keep-alive | HTTP keep-alive |
|------|---------------|-----------------|
| 계층 | 전송 계층 (L4) | 응용 계층 (L7) |
| 목적 | 연결이 살아있는지 **확인** | 하나의 연결로 여러 HTTP 요청 **재사용** |
| 동작 | 주기적으로 프로브 패킷 전송, 무응답 시 연결 종료 | `Connection: keep-alive` 헤더로 요청 |
| 배경 | 유휴 연결의 갑작스러운 종료 감지 | TCP 연결 오버헤드 감소 |

## HTTP 버전 및 성능

### NET-007. HTTP/1.1과 HTTP/2의 차이점은 무엇인가요?

| 구분 | HTTP/1.1 | HTTP/2 |
|------|----------|--------|
| 전송 형식 | 텍스트 | **바이너리 프레임** |
| 멀티플렉싱 | 없음 (파이프라이닝 한계) | **있음** — 하나의 연결에서 여러 스트림 병렬 처리 |
| 헤더 압축 | 없음 | **HPACK** 압축 |
| 서버 푸시 | 없음 | **있음** — 클라이언트 요청 전에 리소스 선제 전송 |
| HOL Blocking | 애플리케이션 레벨 발생 | 애플리케이션 레벨 해결 (TCP 레벨은 여전히 존재) |
| 우선순위 | 없음 | 스트림 우선순위 지정 가능 |

---

### NET-008. HOL Blocking에 대해 설명해 주세요.

HOL(Head-of-Line) Blocking은 **앞에 있는 요청/패킷이 처리될 때까지 뒤의 것들이 대기해야 하는 현상**입니다.

- **HTTP/1.1**: 요청 순서대로 응답해야 하므로, 첫 번째 요청이 느리면 이후 요청 전체가 블로킹됨
- **HTTP/2**: 멀티플렉싱으로 애플리케이션 레벨 HOL Blocking 해결. 그러나 **TCP 레벨**에서는 여전히 발생 — 패킷 하나가 손실되면 모든 스트림이 대기
- **HTTP/3(QUIC)**: UDP 기반 + 스트림 독립성으로 TCP 레벨 HOL Blocking도 해결

---

### NET-009. HTTP/3.0의 주요 특징에 대해 설명해 주세요.

HTTP/3는 QUIC 프로토콜 위에서 동작합니다.

- **QUIC(UDP 기반)**: TCP 대신 UDP 위에서 신뢰성, 순서 보장, 혼잡 제어를 직접 구현
- **연결 수립 단축**: TLS 1.3을 QUIC에 통합 → 0-RTT 또는 1-RTT로 연결 (TCP + TLS 대비 현저히 빠름)
- **HOL Blocking 완전 해결**: 스트림별로 독립적인 패킷 손실 처리
- **연결 마이그레이션**: IP가 변경되어도 **Connection ID**로 연결 유지 (Wi-Fi → LTE 전환 시 유용)
- **내장 암호화**: TLS 1.3이 기본 통합되어 선택이 아닌 필수

---

### NET-010. 왜 HTTP는 TCP를 사용하나요?

HTTP가 TCP를 사용하는 이유는 **신뢰성 있는 데이터 전송** 때문입니다.

- **순서 보장**: 수신 측에서 올바른 순서로 데이터 재조합
- **오류 검출 및 재전송**: 손실된 세그먼트 자동 재전송
- **흐름 제어**: 수신 측이 처리 가능한 속도로 데이터 수신
- **혼잡 제어**: 네트워크 과부하 방지

웹 페이지, API 응답, 파일 등에서 데이터 일부라도 손실되면 심각한 문제가 발생하므로, 신뢰성이 최우선입니다.

---

### NET-011. 그렇다면, 왜 HTTP/3에서는 UDP를 사용하나요? 위에서 언급한 UDP의 문제가 해결되었나요?

HTTP/3는 UDP를 사용하되, **QUIC이 UDP 위에서 TCP의 단점 없이 신뢰성을 직접 구현**합니다.

- QUIC은 UDP 위에서 자체적인 재전송, 순서 보장, 오류 검출, 혼잡 제어를 구현 → UDP의 비신뢰성 문제 해결
- 동시에 TCP의 고질적 문제인 HOL Blocking, 긴 핸드셰이크를 제거
- 즉, "신뢰성은 유지하면서 TCP의 제약을 없앤" 새로운 전송 프로토콜을 UDP 위에 구현한 것입니다.

---

### NET-012. 그런데, 브라우저는 어떤 서버가 TCP를 쓰는지 UDP를 쓰는지 어떻게 알 수 있나요?

`Alt-Svc` HTTP 응답 헤더를 통해 알 수 있습니다.

1. 브라우저는 처음에 TCP(HTTP/1.1 또는 HTTP/2)로 연결
2. 서버가 HTTP 응답에 `Alt-Svc: h3=":443"; ma=86400` 헤더를 포함하여 HTTP/3 지원을 광고
3. 브라우저는 이후 연결 시 QUIC/HTTP/3를 시도
4. 일부 브라우저는 TCP와 UDP 연결을 **병렬로 시도**하여 더 빠른 것을 선택

또한 브라우저 내부에 특정 도메인의 프로토콜 지원 여부를 캐싱해두기도 합니다.

## HTTP 응답 코드

### NET-013. HTTP 응답코드에 대해 설명해 주세요.

HTTP 응답 코드는 서버가 클라이언트의 요청 처리 결과를 3자리 숫자로 전달하는 상태 코드입니다.

| 범위 | 의미 | 주요 코드 |
|------|------|----------|
| **1xx** | Informational (정보) | 100 Continue, 101 Switching Protocols |
| **2xx** | Success (성공) | 200 OK, 201 Created, 204 No Content |
| **3xx** | Redirection (리다이렉션) | 301 Moved Permanently, 302 Found, 304 Not Modified |
| **4xx** | Client Error (클라이언트 오류) | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found |
| **5xx** | Server Error (서버 오류) | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable |

---

### NET-014. 401 (Unauthorized) 와 403 (Forbidden)은 의미적으로 어떤 차이가 있나요?

| 코드 | 의미 | 실제 상황 |
|------|------|----------|
| **401** | **인증(Authentication)** 실패 | 로그인이 안 된 상태. "당신이 누구인지 모름" |
| **403** | **인가(Authorization)** 실패 | 로그인은 됐지만 권한이 없음. "당신이 누구인지는 알지만 접근 불가" |

- 401의 이름이 "Unauthorized"이지만 사실상 **Unauthenticated**의 의미
- 403은 신원은 확인됐으나 해당 자원에 대한 **권한 없음**
- 예: 미로그인 사용자가 마이페이지 접근 → 401 / 일반 사용자가 관리자 페이지 접근 → 403

---

### NET-015. 200 (ok) 와 201 (created) 의 차이에 대해 설명해 주세요.

- **200 OK**: 요청이 성공적으로 처리됨. GET, PUT, DELETE 등 일반적인 성공 응답
- **201 Created**: 요청이 성공적으로 처리되어 **새 리소스가 생성**됨. 주로 POST 요청에 사용
  - 응답에 `Location: /users/123` 헤더로 생성된 자원의 URI를 포함하는 것이 관례

RESTful API에서 명확한 상태 코드 사용은 클라이언트가 응답 결과를 올바르게 해석하는 데 중요합니다.

---

### NET-016. 필요하다면 저희가 직접 응답코드를 정의해서 사용할 수 있을까요? 예를 들어 285번 처럼요.

기술적으로는 가능하지만 **권장하지 않습니다**.

- HTTP 명세(RFC 7231)는 표준에 없는 코드의 사용을 완전히 금지하지 않으며, 미등록 코드도 해당 범위(1xx~5xx)의 의미를 따라야 함
- 285는 2xx 범위이므로 "성공"의 의미를 가지게 됨
- **문제점**: 프록시, 캐시, 브라우저, 라이브러리가 커스텀 코드를 이해하지 못해 예기치 않은 동작 발생 가능
- **더 나은 대안**: 응답 바디에 별도의 비즈니스 에러 코드를 정의하는 방식

```json
{
  "code": "USER_NOT_FOUND",
  "message": "해당 사용자를 찾을 수 없습니다."
}
```

## HTTP Method

### NET-017. HTTP Method에 대해 설명해 주세요.

HTTP Method는 클라이언트가 서버에 **요청의 목적**을 알려주는 동사입니다.

| 메서드 | 역할 | 멱등 | 안전 |
|--------|------|------|------|
| GET | 리소스 조회 | O | O |
| POST | 리소스 생성 | X | X |
| PUT | 리소스 전체 교체 | O | X |
| PATCH | 리소스 일부 수정 | △ | X |
| DELETE | 리소스 삭제 | O | X |
| HEAD | GET과 동일하나 바디 없음 | O | O |
| OPTIONS | 서버가 지원하는 메서드 조회 | O | O |

---

### NET-018. HTTP Method의 멱등성에 대해 설명해 주세요.

멱등성(Idempotent)이란 **동일한 요청을 여러 번 전송해도 서버의 상태가 동일한 것**을 의미합니다.

- **멱등**: GET, PUT, DELETE, HEAD, OPTIONS
  - GET: 여러 번 조회해도 서버 상태 불변
  - PUT: 같은 데이터로 여러 번 교체해도 결과 동일
  - DELETE: 이미 삭제된 리소스 재삭제 시도해도 결과(리소스 없음) 동일
- **비멱등**: POST, PATCH
  - POST: 여러 번 호출 시 여러 리소스 생성
  - PATCH: 구현에 따라 다름 (값을 특정값으로 **설정** → 멱등 / 값을 **증가** → 비멱등)

멱등성은 네트워크 오류 시 **재시도(Retry) 전략**을 수립할 때 핵심 기준이 됩니다.

---

### NET-019. GET과 POST의 차이는 무엇인가요?

| 구분 | GET | POST |
|------|-----|------|
| 목적 | 데이터 조회 | 데이터 생성/전송 |
| 데이터 위치 | URL 쿼리스트링 | 요청 바디 |
| 캐싱 | 가능 | 기본적으로 불가 |
| 보안 | URL에 노출됨 | 바디에 위치 (상대적 안전) |
| 멱등성 | 멱등 | 비멱등 |
| 북마크 | 가능 | 불가 |
| 데이터 크기 | URL 길이 제한 | 이론적으로 제한 없음 |

---

### NET-020. POST와 PUT, PATCH의 차이는 무엇인가요?

| 구분 | POST | PUT | PATCH |
|------|------|-----|-------|
| 역할 | 리소스 **생성** | 리소스 **전체 교체** | 리소스 **일부 수정** |
| URI | 서버가 결정 (`/users` → `/users/123`) | 클라이언트가 지정 (`/users/123`) | 클라이언트가 지정 |
| 없는 경우 | 새로 생성 | 생성하거나 교체 | 오류 처리 |
| 멱등성 | X | O | △ |

- PUT은 전체 필드를 보내야 하며, 보내지 않은 필드는 null이 됨
- PATCH는 수정할 필드만 전송

---

### NET-021. HTTP 1.1 이후로, GET에도 Body에 데이터를 실을 수 있게 되었습니다. 그럼에도 불구하고 왜 아직도 이런 방식을 지양하는 것일까요?

여러 현실적 이유 때문입니다.

- **HTTP 명세(RFC 7231)**: GET 바디는 "정의된 의미가 없다"고 명시 — 구현체가 무시해도 문제없음
- **인프라 호환성 문제**: 많은 프록시, 캐시, WAF, API Gateway가 GET 바디를 무시하거나 제거
- **캐싱 불가**: 캐시 인프라가 GET 바디를 캐시 키로 포함하지 않아 예측 불가능한 동작 발생
- **가독성 및 디버깅**: URL 쿼리스트링이 로그와 디버깅에서 훨씬 직관적
- **상호운용성(Interoperability)**: 표준 준수 클라이언트/서버 간 일관된 동작 보장이 어려움

---

## 쿠키와 세션

### NET-022. 쿠키와 세션의 차이에 대해 설명해 주세요.

| 구분 | 쿠키(Cookie) | 세션(Session) |
|------|-------------|--------------|
| 저장 위치 | **클라이언트** (브라우저) | **서버** |
| 보안 | 상대적으로 취약 (클라이언트 노출) | 상대적으로 안전 (서버 보관) |
| 용량 | 약 4KB 제한 | 서버 메모리에 따라 다름 |
| 만료 | 설정한 시간 기반 (영속 가능) | 세션 종료 또는 타임아웃 |
| 속도 | 빠름 (서버 조회 불필요) | 서버 조회 필요 |
| 사용 예 | 자동 로그인, 설정 저장 | 로그인 상태 유지 |

---

### NET-023. 세션 방식의 로그인 과정에 대해 설명해 주세요.

1. 클라이언트가 아이디/비밀번호로 **로그인 요청**
2. 서버가 자격증명 검증 후 **세션 생성** → 세션 저장소(메모리, DB, Redis)에 저장
3. 서버가 `Set-Cookie: sessionId=abc123` 응답 헤더로 **세션 ID 전달**
4. 이후 클라이언트는 모든 요청에 `Cookie: sessionId=abc123` **헤더를 자동 포함**
5. 서버는 세션 ID로 세션 저장소를 조회하여 사용자 식별
6. 로그아웃 시 서버에서 **세션 삭제**

---

### NET-024. Stateless의 의미를 살펴보면, 세션은 적절하지 않은 인증 방법 아닌가요?

엄밀히 말하면 **세션은 Stateless 원칙을 위반**합니다.

- 서버가 클라이언트의 상태(세션 정보)를 저장하므로 Stateful
- 그러나 이는 HTTP의 Connectionless/Stateless를 보완하기 위한 **현실적 트레이드오프**
- 완전한 Stateless 인증 방법으로 **JWT(JSON Web Token)**가 있음
  - 서버가 상태를 저장하지 않고, 토큰 자체에 사용자 정보를 담아 서명
  - 단, 토큰 무효화가 어려운 단점 존재 (로그아웃 처리 등)
- 결론: 세션은 보안성 확보를 위해 Stateless 원칙을 의도적으로 타협한 방식

---

### NET-025. 규모가 커져 서버가 여러 개가 된다면, 세션을 어떻게 관리할 수 있을까요?

세 가지 주요 전략이 있습니다.

1. **Sticky Session(세션 고정)**: 로드밸런서가 특정 클라이언트 요청을 항상 같은 서버로 라우팅
   - 장점: 구현 단순 / 단점: 특정 서버 부하 집중, 서버 장애 시 세션 소실

2. **세션 클러스터링**: 모든 서버 간 세션 정보를 실시간 동기화
   - 장점: 서버 장애 대비 / 단점: 서버 간 네트워크 트래픽 증가, 복잡도 증가

3. **중앙 집중식 세션 저장소**: **Redis, Memcached** 같은 외부 저장소에 세션 저장, 모든 서버가 공유 조회
   - 가장 많이 사용되는 방식. Scale-out과 장애 내성 모두 확보

4. **JWT 전환**: Stateless 인증으로 세션 문제를 근본적으로 해결

## HTTPS 및 보안

### NET-026. 공개키와 대칭키에 대해 설명해 주세요.

**대칭키(Symmetric Key)**
- 암호화와 복호화에 **같은 키** 사용
- 빠른 처리 속도
- 문제: 키를 안전하게 공유하는 방법이 필요 (Key Distribution Problem)
- 예: AES, DES

**공개키/비대칭키(Asymmetric Key)**
- **공개키(Public Key)**로 암호화 → **개인키(Private Key)**로만 복호화
- 공개키는 누구에게나 공개, 개인키는 본인만 보유
- 키 배포 문제 해결, 처리 속도 느림
- 예: RSA, ECC

**실제 HTTPS**: 공개키로 대칭키를 안전하게 교환한 뒤, 실제 데이터는 대칭키로 암호화 (속도와 보안 모두 확보)

---

### NET-027. 왜 HTTPS Handshake 과정에서는 인증서를 사용하는 것 일까요?

**서버의 신뢰성(신원)을 검증**하기 위해서입니다.

공개키만으로는 "이 공개키가 진짜 그 서버의 것인지" 알 수 없어 **중간자 공격(MITM)**에 취약합니다.

- 인증서는 신뢰할 수 있는 제3자 **CA(Certificate Authority)**가 서버의 신원과 공개키를 보증하는 문서
- 브라우저는 CA의 루트 인증서를 사전에 신뢰 목록에 보유
- 서버가 위조된 공개키를 제공해도 CA 서명이 없으면 브라우저가 경고

인증서 없이 공개키만 사용하면, 공격자가 중간에서 자신의 공개키를 서버인 척 제공하고 트래픽을 복호화할 수 있습니다.

---

### NET-028. SSL과 TLS의 차이는 무엇인가요?

| 구분 | SSL | TLS |
|------|-----|-----|
| 개발사 | Netscape | IETF (표준화) |
| 버전 | SSL 1.0 ~ 3.0 | TLS 1.0 ~ 1.3 |
| 상태 | 모든 버전 deprecated | TLS 1.2, 1.3 현재 사용 |

- TLS 1.0은 SSL 3.1에서 발전한 후속 표준 (TLS = SSL 4.0이라고도 볼 수 있음)
- 현재 SSL은 POODLE, BEAST 등 취약점으로 인해 모든 버전이 사용 금지
- 실제로 "SSL 인증서", "SSL 통신"이라고 말해도 대부분 **TLS**를 의미하는 관행적 표현
- **TLS 1.3** (2018): 구형 암호 알고리즘 제거, 핸드셰이크 1-RTT로 단축, 0-RTT 지원

---

### NET-029. XSS에 대해서 설명해 주세요.

XSS(Cross-Site Scripting)는 **공격자가 웹 페이지에 악성 스크립트를 삽입하여 다른 사용자의 브라우저에서 실행**되도록 하는 공격입니다.

| 유형 | 설명 |
|------|------|
| **Stored XSS** | 악성 스크립트를 DB에 저장 → 다른 사용자 조회 시 실행 |
| **Reflected XSS** | URL에 포함된 스크립트가 서버 응답에 반사되어 실행 |
| **DOM-based XSS** | 서버를 거치지 않고 클라이언트 측 DOM 조작으로 발생 |

**피해**: 세션 쿠키 탈취, 키로깅, 피싱 페이지 삽입, 사용자 정보 유출

**방어**: 입력값 검증·이스케이핑, Content Security Policy(CSP), HttpOnly 쿠키

---

### NET-030. CSRF랑 XSS는 어떤 차이가 있나요?

| 구분 | XSS | CSRF |
|------|-----|------|
| 공격 방식 | 공격자 스크립트가 피해자 브라우저에서 실행 | 피해자가 의도치 않게 요청을 전송 |
| 신뢰 악용 | 사용자가 사이트를 신뢰하는 것 악용 | **서버가 사용자를 신뢰**하는 것 악용 |
| 목적 | 데이터 탈취, 세션 하이재킹 | 피해자 권한으로 서버에 악의적 요청 |
| 방어 | CSP, 입력값 이스케이핑 | CSRF 토큰, SameSite 쿠키 |

CSRF는 이미 인증된 사용자의 브라우저를 이용해 사용자 모르게 서버에 요청을 전송하는 공격입니다. 예: 로그인된 사용자가 악성 링크를 클릭하면 자신의 계정으로 송금 요청이 전송.

---

### NET-031. XSS는 프론트엔드에서만 막을 수 있나요?

아니요, **다층 방어(Defense in Depth)**가 필요합니다.

- **프론트엔드**: 입력값 이스케이핑, `innerHTML` 대신 `textContent` 사용, 신뢰할 수 없는 데이터 직접 삽입 금지
- **백엔드**: 서버 측 입력 검증 및 이스케이핑 — **필수**. 프론트엔드는 우회 가능하므로 백엔드 검증이 최후 방어선
- **HTTP 헤더**:
  - `Content-Security-Policy`: 허용된 스크립트 출처 제한
  - `HttpOnly`: 쿠키를 JavaScript에서 접근 불가 → 세션 탈취 방지
  - `X-Content-Type-Options`: MIME 타입 스니핑 방지

결론: **프론트엔드는 보조 역할**, 백엔드 검증 없이 XSS를 막는 것은 불가능합니다.

## CORS 및 보안 정책

### NET-032. SOP 정책에 대해 설명해 주세요.

SOP(Same-Origin Policy, 동일 출처 정책)는 **브라우저가 한 출처에서 로드된 문서나 스크립트가 다른 출처의 리소스와 상호작용하는 것을 제한**하는 보안 정책입니다.

**출처(Origin) = 프로토콜 + 호스트 + 포트** 세 가지가 모두 같아야 동일 출처.

| 비교 대상 | 결과 |
|----------|------|
| `https://a.com:443` vs `http://a.com:443` | 다름 (프로토콜) |
| `https://a.com` vs `https://b.com` | 다름 (호스트) |
| `https://a.com:443` vs `https://a.com:8080` | 다름 (포트) |

- 목적: XSS, CSRF 등의 공격 방지
- 제한 대상: XHR/Fetch API, DOM 접근, 쿠키
- 허용 예외: `<img>`, `<script src>`, `<link>` 태그를 통한 임베딩은 허용

---

### NET-033. CORS 정책이 무엇인가요?

CORS(Cross-Origin Resource Sharing)는 **SOP를 완화하여 다른 출처의 리소스에 접근할 수 있도록 서버가 명시적으로 허용**하는 메커니즘입니다.

- 서버가 HTTP 응답 헤더에 허용할 출처를 명시
  - `Access-Control-Allow-Origin: https://example.com`
  - `Access-Control-Allow-Methods: GET, POST, PUT`
- 브라우저는 실제 요청 전(또는 응답 후) 이 헤더를 확인하여 허용 여부 결정
- **CORS는 브라우저 정책** → 서버-서버 통신에는 적용되지 않음
- 브라우저가 시행하는 정책이므로 서버는 CORS 헤더만 올바르게 설정하면 됨

---

### NET-034. Preflight에 대해 설명해 주세요.

Preflight는 브라우저가 **실제 요청 전에 OPTIONS 메서드로 서버에 허용 여부를 먼저 확인**하는 과정입니다.

**단순 요청(Preflight 생략)**:
- GET, HEAD, POST (단, 특정 Content-Type만)
- 커스텀 헤더 없음

**Preflight가 필요한 경우**:
- PUT, DELETE, PATCH 메서드
- `Content-Type: application/json` 등 비표준 타입
- 커스텀 헤더 포함

**과정**:
1. 브라우저 → 서버: `OPTIONS /api/users` (Origin, Access-Control-Request-Method 등 포함)
2. 서버 → 브라우저: 허용 정보 응답 (Access-Control-Allow-Origin, Allow-Methods 등)
3. 허용 확인 후 → 브라우저가 실제 요청 전송

`Access-Control-Max-Age` 헤더로 Preflight 결과를 캐시하여 반복 요청을 줄일 수 있습니다.

## TCP / UDP

### NET-035. TCP와 UDP의 차이에 대해 설명해 주세요.

| 구분 | TCP | UDP |
|------|-----|-----|
| 연결 | 연결 지향 (3-way handshake) | 비연결 |
| 신뢰성 | 보장 (재전송, 순서 보장) | 미보장 |
| 속도 | 상대적으로 느림 | 빠름 |
| 흐름 제어 | 있음 | 없음 |
| 혼잡 제어 | 있음 | 없음 |
| 헤더 크기 | 20~60바이트 | 8바이트 |
| 주요 사용 | 웹, 이메일, 파일 전송 | 실시간 스트리밍, DNS, 게임 |

---

### NET-036. Checksum이 무엇인가요?

Checksum은 **데이터 전송 오류를 검출하기 위해 데이터의 내용을 기반으로 계산된 값**입니다.

- **송신 측**: 데이터의 각 바이트를 합산(또는 CRC 등 알고리즘 적용)하여 Checksum 계산 → 헤더에 포함하여 전송
- **수신 측**: 수신된 데이터로 Checksum 재계산 → 헤더의 값과 비교
- 불일치 시: 데이터 손상(오류) 감지

---

### NET-037. TCP와 UDP 중 어느 프로토콜이 Checksum을 수행할까요?

**두 프로토콜 모두** Checksum 필드를 헤더에 포함합니다.

- **TCP**: Checksum 필수. 항상 계산하여 헤더에 포함
- **UDP**: Checksum 필드 존재. IPv4에서는 **선택적** (0으로 설정 가능), IPv6에서는 **필수**

---

### NET-038. 그렇다면, Checksum을 통해 오류를 정정할 수 있나요?

아니요. Checksum은 오류 **검출(Detection)**만 가능하고 **정정(Correction)은 불가**합니다.

- Checksum은 오류의 존재를 알려주지만, 어느 비트가 잘못됐는지 알 수 없음
- 오류 발견 시:
  - TCP: 해당 세그먼트 폐기 후 재전송 요청 (ACK 미전송)
  - UDP: 세그먼트 폐기 (재전송 없음)
- 오류 **정정**이 필요하면 FEC(Forward Error Correction) 같은 별도 메커니즘 사용 (예: QUIC, 위성통신)

---

### NET-039. TCP가 신뢰성을 보장하는 방법에 대해 설명해 주세요.

| 메커니즘 | 설명 |
|---------|------|
| **시퀀스 번호** | 패킷에 순서 번호 부여 → 순서 재조립, 중복 감지 |
| **확인 응답(ACK)** | 수신 측이 받은 데이터를 확인 → 미수신 시 재전송 |
| **재전송** | ACK 타임아웃 또는 3번 중복 ACK 수신 시 자동 재전송 |
| **흐름 제어** | 수신 윈도우(rwnd) 기반으로 수신 측 처리 능력에 맞춰 전송 속도 조절 |
| **혼잡 제어** | 네트워크 혼잡 감지 시 전송 속도 감소 |
| **Checksum** | 데이터 무결성 검증 |

---

### NET-040. TCP의 혼잡 제어 처리 방법에 대해 설명해 주세요.

TCP 혼잡 제어는 **cwnd(혼잡 윈도우)**와 **ssthresh(슬로우스타트 임계값)**를 기반으로 동작합니다.

- **Slow Start**: 연결 초기 cwnd=1 MSS에서 시작, ACK마다 지수적 증가 (1→2→4→8...)
- **Congestion Avoidance**: ssthresh 도달 후 RTT마다 1 MSS씩 선형 증가
- **Fast Retransmit**: 3개의 중복 ACK 수신 시 타임아웃 전에 즉시 재전송
- **Fast Recovery**: Fast Retransmit 후 Slow Start를 건너뛰고 Congestion Avoidance로 진입

**패킷 손실 감지 시 반응**:
- 타임아웃: ssthresh = cwnd/2, cwnd = 1 MSS → Slow Start 재시작
- 3 중복 ACK: ssthresh = cwnd/2, cwnd = ssthresh (TCP Reno 기준)

---

### NET-041. 본인이 새로운 통신 프로토콜을 TCP나 UDP를 사용해서 구현한다고 하면, 어떤 기준으로 프로토콜을 선택하시겠어요?

**데이터 무결성 vs 실시간성**이 핵심 기준입니다.

**TCP를 선택하는 경우**:
- 데이터 손실이 허용되지 않는 경우 (금융 거래, 파일 전송, 이메일)
- 순서 보장이 중요한 경우
- 신뢰성 메커니즘을 직접 구현하지 않고 빠르게 개발해야 하는 경우

**UDP를 선택하는 경우**:
- 실시간성이 최우선인 경우 (화상통화, 실시간 게임, 라이브 스트리밍)
- 일부 패킷 손실이 허용되는 경우 (오래된 프레임보다 최신 프레임이 더 중요)
- 낮은 레이턴시가 필요한 경우
- 직접 신뢰성 메커니즘을 구현하여 더 세밀하게 제어하고 싶은 경우 (QUIC처럼)

핵심 질문: **"패킷 하나의 손실이 치명적인가?"** → TCP / **"약간의 손실보다 지연이 더 나쁜가?"** → UDP

## 3-Way / 4-Way Handshake

### NET-042. 3-Way Handshake에 대해 설명해 주세요.

TCP 연결 수립 과정입니다.

```
클라이언트                    서버
    |                          |
    |------- SYN(seq=x) ------>|  1단계: 연결 요청
    |                          |
    |<-- SYN-ACK(seq=y,ack=x+1)|  2단계: 연결 수락 + 상대 시퀀스 확인
    |                          |
    |------ ACK(ack=y+1) ----->|  3단계: 연결 확정
    |                          |
```

- **SYN**: 클라이언트가 임의의 시퀀스 번호 X로 연결 요청
- **SYN-ACK**: 서버가 X+1로 ACK하고, 자신의 시퀀스 번호 Y 전송
- **ACK**: 클라이언트가 Y+1로 ACK → **연결 수립 완료**

목적: 양방향 통신 채널 확보 + 양측 시퀀스 번호 동기화

---

### NET-043. ACK, SYN 같은 정보는 어떻게 전달하는 것 일까요?

TCP 세그먼트 **헤더의 제어 플래그(Control Flags)** 비트를 통해 전달됩니다.

TCP 헤더에는 6개의 1비트 플래그 필드가 있습니다:

| 플래그 | 의미 |
|--------|------|
| **SYN** | 연결 요청 |
| **ACK** | 확인 응답 (Acknowledgment Number 유효) |
| **FIN** | 연결 종료 요청 |
| **RST** | 연결 강제 리셋 |
| **PSH** | 수신 버퍼 즉시 전달 |
| **URG** | 긴급 데이터 포함 |

각 플래그를 0 또는 1로 설정하여 패킷의 목적을 표시하며, Sequence Number와 Acknowledgment Number 필드도 함께 사용됩니다.

---

### NET-044. 2-Way Handshaking을 하지않는 이유에 대해 설명해 주세요.

2-way로는 **양방향 연결 확립이 불가능**하기 때문입니다.

- 1단계 SYN: 클라이언트 → 서버 도달 가능한지 확인
- 2단계 SYN-ACK: 서버 → 클라이언트 도달 가능한지 확인 + 클라이언트 SYN 확인
- 3단계 ACK: **클라이언트가 서버의 SYN-ACK를 실제로 받았음**을 서버가 확인

2-way에서는 서버가 자신의 SYN에 대한 ACK를 받지 못해 **서버 → 클라이언트 방향의 통신 가능 여부를 확인할 수 없습니다**. 또한 양측 시퀀스 번호를 동기화하려면 3-way가 필요합니다.

---

### NET-045. 두 호스트가 동시에 연결을 시도하면, 연결이 가능한가요? 가능하다면 어떻게 통신 연결을 수행하나요?

가능합니다. 이를 **Simultaneous Open(동시 개방)**이라 합니다.

```
호스트 A                      호스트 B
    |------- SYN(seq=x) ------>|
    |<------ SYN(seq=y) -------|
    |<---- SYN-ACK(ack=x+1) ---|
    |--- SYN-ACK(ack=y+1) ---->|
    |          (연결 수립)      |
```

- 일반 3-way: 3개 패킷 교환 / 동시 개방: 4개 패킷 교환
- TCP 명세에 이 케이스가 명시되어 있으며, 정상적으로 연결이 수립됩니다.

---

### NET-046. SYN Flooding에 대해 설명해 주세요.

SYN Flooding은 **TCP 3-way handshake의 취약점을 이용한 DoS/DDoS 공격**입니다.

**공격 과정**:
1. 공격자가 **위조된(존재하지 않는) 출발지 IP**로 대량의 SYN 패킷 전송
2. 서버는 SYN-ACK 응답 후 ACK를 기다리며 **Half-Open 연결을 백로그 큐에 저장**
3. 가짜 IP이므로 ACK가 돌아오지 않음 → 큐가 가득 차 정상 연결 불가

**방어**:
- **SYN Cookies**: 백로그 큐 없이 SYN-ACK에 쿠키 값 삽입, ACK 수신 시 검증 → 큐 고갈 방지
- 방화벽/IPS에서 동일 IP의 SYN 급증 탐지 및 차단
- Rate Limiting으로 동일 IP당 SYN 속도 제한

---

### NET-047. 3-Way Handshake의 속도 문제 때문에 이동 수를 줄이는 0-RTT 기법을 많이 적용하고 있습니다. 어떤 방식으로 가능한 걸까요?

이전 연결 정보를 재사용하는 방식으로 핸드셰이크를 생략합니다.

- **TLS 1.3의 1-RTT**: 이전 TLS 버전(2-RTT)보다 핸드셰이크 라운드트립을 1번으로 단축
- **TLS 1.3의 0-RTT(Early Data)**:
  - 이전 세션에서 받은 **PSK(Pre-Shared Key)** 또는 세션 티켓을 저장
  - 다음 연결 시 첫 번째 패킷에 바로 데이터 포함
  - 단, **Replay Attack**에 취약 → 안전하지 않은 POST 요청 등에는 주의 필요
- **QUIC의 0-RTT**:
  - TCP 연결 + TLS 핸드셰이크를 통합하여 이전 연결 정보를 재사용
  - 초기 연결 1-RTT, 재연결 0-RTT

---

### NET-048. 4-Way Handshake에 대해 설명해 주세요.

TCP 연결 종료 과정입니다.

```
클라이언트                    서버
    |------- FIN ------------>|  1단계: 종료 요청 (더 이상 보낼 데이터 없음)
    |<------ ACK -------------|  2단계: FIN 확인 (서버는 아직 데이터 전송 가능 → Half-Close)
    |                          |  (서버가 남은 데이터 전송)
    |<------ FIN -------------|  3단계: 서버도 종료 요청
    |------- ACK ------------>|  4단계: 서버 FIN 확인
    |   (TIME_WAIT 대기)       |
```

3-way와 달리 4-way인 이유: 서버가 FIN을 받아도 아직 전송할 데이터가 남아있을 수 있어 ACK와 FIN을 분리합니다.

---

### NET-049. 패킷이 4-way handshake 목적인지 어떻게 파악할 수 있을까요?

TCP 헤더의 **FIN 플래그 비트**로 구분합니다.

- FIN 플래그 = 1인 패킷 → 연결 종료 목적
- ACK 플래그와 함께 사용 (FIN+ACK)
- (출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트) 4-tuple로 어느 연결에 대한 종료인지 식별
- Wireshark 같은 패킷 분석 도구에서는 `TCP FIN` 또는 `[FIN, ACK]`로 표시

---

### NET-050. 빨리 끊어야 할 경우엔, (즉, 4-way Handshake를 할 여유가 없다면) 어떻게 종료할 수 있을까요?

**RST(Reset) 플래그**를 사용합니다.

- RST=1 패킷을 전송하면 4-way 없이 **즉시 연결을 강제 종료**
- 수신 측은 연결이 비정상 종료되었음을 즉시 인식
- **사용 사례**: 오류 상황, 보안 위협 감지(방화벽), 타임아웃, 프로그램 크래시
- FIN과 달리 RST는 정상적인 종료가 아니므로, 미전송 데이터는 유실될 수 있음

---

### NET-051. 4-Way Handshake 과정에서 중간에 한쪽 네트워크가 강제로 종료된다면, 반대쪽은 이를 어떻게 인식할 수 있을까요?

즉각적인 감지는 어려우며 다음 메커니즘으로 인식합니다.

- **TCP Keep-Alive**: 일정 시간 데이터가 없으면 프로브 패킷 전송. 무응답 시 연결 종료
- **타임아웃**: FIN/ACK에 대한 응답이 없으면 재전송 타임아웃 후 연결 종료
- **RST 수신**: 중간 네트워크 장비(방화벽 등)가 RST를 보내 통보
- **데이터 전송 시**: 실제로 데이터를 보내면 오류가 발생해 감지
- 유휴(Idle) 연결에서는 Keep-Alive 없이 즉시 감지가 어려워 "좀비 연결" 상태가 됩니다.

---

### NET-052. 왜 종료 후에 바로 끝나지 않고, TIME_WAIT 상태로 대기하는 것 일까요?

두 가지 중요한 이유가 있습니다.

1. **마지막 ACK 보장**: 클라이언트가 서버의 FIN에 대한 ACK를 전송했지만, 이 ACK가 손실될 수 있습니다. 서버가 FIN을 재전송할 때 클라이언트가 응답할 수 있도록 대기합니다.

2. **지연 패킷 처리**: 이전 연결의 패킷이 네트워크에 늦게 도착할 수 있습니다. 동일한 4-tuple로 새 연결이 생성됐을 때 이전 패킷과 혼동하는 것을 방지합니다.

- **TIME_WAIT 시간**: 2 MSL (Maximum Segment Lifetime, 일반적으로 60초~4분)
- TIME_WAIT가 너무 많으면 포트 고갈 문제가 발생할 수 있어 `SO_REUSEADDR` 소켓 옵션으로 완화 가능

## 소켓 통신

### NET-053. 웹소켓과 소켓 통신의 차이에 대해 설명해 주세요.

| 구분 | 일반 소켓 (TCP/UDP) | 웹소켓 (WebSocket) |
|------|-------------------|------------------|
| 수준 | OS 레벨 네트워크 인터페이스 | 애플리케이션 계층 프로토콜 |
| 기반 | TCP 또는 UDP | TCP (HTTP 위에서 동작) |
| 시작 | 직접 연결 | HTTP 핸드셰이크 후 프로토콜 업그레이드 |
| 통신 | 양방향 | 양방향 |
| 브라우저 | 직접 사용 불가 | 브라우저에서 직접 사용 가능 |
| 포트 | 임의 | 80(ws://) / 443(wss://) |
| 사용처 | 모든 네트워크 애플리케이션 | 채팅, 실시간 알림, 게임 |

웹소켓은 TCP 소켓 위에 구현된 **애플리케이션 계층 프로토콜**입니다.

---

### NET-054. 소켓과 포트의 차이가 무엇인가요?

- **포트(Port)**: 논리적인 **번호**. 같은 IP에서 여러 프로세스를 구분하는 식별자 (0~65535)
- **소켓(Socket)**: 실제 통신을 위한 **엔드포인트**. IP 주소 + 포트 번호의 조합

더 정확히는, TCP 소켓은 다음 4-tuple로 완전히 식별됩니다:

```
(출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트)
```

포트는 번호이고, 소켓은 그 포트를 포함하는 **통신 엔드포인트**입니다.

---

### NET-055. 여러 소켓이 있다고 할 때, 그 소켓의 포트 번호는 모두 다른가요?

아니요. **서버 측에서는 여러 소켓이 같은 포트를 공유**할 수 있습니다.

- 서버 포트 80으로 1000명의 클라이언트가 연결되면, 서버에는 1000개의 소켓이 생성
- 모든 소켓의 목적지 포트는 80이지만, 각 소켓의 **(출발지 IP, 출발지 포트)가 다름** → 4-tuple이 유일
- 운영체제는 이 4-tuple로 각 소켓을 구분하여 정확한 연결로 데이터를 전달합니다.

---

### NET-056. 사용자의 요청이 무수히 많아지면, 소켓도 무수히 생성되나요?

기본적으로는 맞지만, 시스템 한계와 최적화가 있습니다.

- 운영체제의 **파일 디스크립터 제한** (`ulimit -n`, 기본값 1024 또는 65536)에 의해 최대 소켓 수 제한
- 소켓마다 메모리와 파일 디스크립터가 필요

**해결 방법**:
- **비동기 I/O (epoll, kqueue)**: 하나의 스레드로 수만 개의 소켓을 이벤트 기반으로 처리 (C10K 문제 해결)
- **Connection Pooling**: 연결을 미리 생성하고 재사용
- **Keep-Alive**: 기존 연결 재사용으로 새 소켓 생성 최소화
- **로드밸런서**: 여러 서버에 연결을 분산

---

### NET-057. 멀티플렉싱과 디멀티플렉싱에 대해 설명해 주세요.

- **멀티플렉싱(Multiplexing)**: 여러 소켓의 데이터를 하나의 네트워크 채널로 합쳐 전송
  - 송신 측 전송 계층에서 여러 프로세스의 데이터를 세그먼트로 묶어 전송
  - 포트 번호를 헤더에 기록하여 수신 측에서 구분 가능하게 함

- **디멀티플렉싱(Demultiplexing)**: 수신된 데이터를 포트 번호 기반으로 적절한 소켓에 전달
  - 수신 측 전송 계층에서 세그먼트의 포트 번호를 확인하여 해당 소켓/프로세스에 전달

---

### NET-058. 디멀티플렉싱의 과정에 대해 설명해 주세요.

**UDP 디멀티플렉싱**: `(목적지 IP, 목적지 포트)`만으로 소켓 식별
- 출발지가 달라도 같은 목적지 포트이면 동일 소켓으로 전달 가능

**TCP 디멀티플렉싱**: `(출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트)` 4-tuple 전체로 소켓 식별
- 같은 목적지 포트여도 출발지가 다르면 **다른 소켓으로 전달**

**상세 과정**:
1. 수신된 세그먼트에서 목적지 포트(TCP는 4-tuple) 추출
2. 운영체제의 소켓 테이블에서 해당 소켓 검색
3. 해당 소켓의 수신 버퍼에 데이터 전달
4. 소켓을 기다리는 프로세스(블로킹 `recv()` 등)에 데이터 제공
